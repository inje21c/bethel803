# 웹푸시 DB 스키마 초안

## 1. 문서 목적

이 문서는 `벧엘구역`의 설치형 앱 및 구독형 알림 기능을 위해 필요한 웹푸시 데이터 구조 초안을 정의한다.

이 문서는 다음을 목표로 한다.

- 웹푸시 구독 저장 구조를 명확히 한다.
- 앱 내 알림과 웹푸시의 관계를 정리한다.
- 발송 결과 추적과 사용자 알림 설정 구조를 정의한다.
- 이후 Supabase 마이그레이션 작성의 기준 문서가 된다.

## 2. 설계 원칙

- 기존 `notifications`를 단일 진실 공급원으로 유지한다.
- 웹푸시는 `notifications`를 외부 채널로 전달하는 구조로 설계한다.
- 구독 정보는 사용자 단위이되, 구역 컨텍스트를 함께 저장한다.
- 발송 결과는 로그로 남겨 운영 중 실패 원인을 추적할 수 있게 한다.
- 알림 선호도는 기능별로 관리하고, 향후 일일/주간 요약 확장에 대응한다.

## 3. 권장 구조 개요

```text
앱 이벤트 발생
 -> notifications insert
 -> 발송 대상 규칙 계산
 -> push_subscriptions 조회
 -> push_deliveries 생성/업데이트
 -> 웹푸시 전송
 -> 성공/실패 로그 저장
```

## 4. 기존 테이블과의 관계

현재 관련 기존 테이블:

- `notifications`
- `notification_reads`
- `users`
- `districts`

신규 권장 테이블:

- `push_subscriptions`
- `notification_preferences`
- `push_deliveries`
- `sermon_drafts` 또는 `content_drafts` (AI 확장용, 선택)

## 5. 신규 테이블 초안

## 5.1 `push_subscriptions`

목적:

- 사용자 브라우저 또는 설치형 앱의 웹푸시 구독 정보를 저장한다.

권장 컬럼:

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `id` | UUID PK | 구독 식별자 |
| `user_id` | UUID FK -> `users.id` | 구독 소유 사용자 |
| `district_id` | UUID FK -> `districts.id` | 구독 당시 소속 구역 |
| `endpoint` | TEXT UNIQUE | Push endpoint |
| `p256dh` | TEXT | Push 공개키 |
| `auth` | TEXT | Push auth secret |
| `platform` | TEXT | `android-chrome`, `ios-webapp`, `desktop-chrome` 등 |
| `user_agent` | TEXT NULL | 디버깅용 브라우저 정보 |
| `app_version` | TEXT NULL | 앱 버전 또는 manifest 버전 |
| `is_active` | BOOLEAN | 현재 유효한 구독 여부 |
| `last_seen_at` | TIMESTAMPTZ NULL | 마지막 활성 확인 시각 |
| `created_at` | TIMESTAMPTZ | 생성 시각 |
| `updated_at` | TIMESTAMPTZ | 수정 시각 |

권장 제약:

- `endpoint` unique
- `user_id` not null
- `district_id` not null
- `is_active` default true

권장 인덱스:

- `(user_id, is_active)`
- `(district_id, is_active)`

메모:

- 사용자가 같은 계정으로 여러 기기에서 구독할 수 있으므로 `user_id` unique는 두지 않는다.

## 5.2 `notification_preferences`

목적:

- 사용자의 푸시 수신 선호도를 저장한다.

권장 컬럼:

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `user_id` | UUID PK/FK -> `users.id` | 사용자 식별자 |
| `schedule_enabled` | BOOLEAN | 일정 알림 수신 여부 |
| `study_enabled` | BOOLEAN | 성경공부 알림 수신 여부 |
| `devotional_enabled` | BOOLEAN | 오늘의 묵상 알림 수신 여부 |
| `prayer_enabled` | BOOLEAN | 중보기도 알림 수신 여부 |
| `reading_weekly_enabled` | BOOLEAN | 주간 성경읽기 리마인더 수신 여부 |
| `service_notice_enabled` | BOOLEAN | 앱 서비스 운영 공지 수신 여부 |
| `quiet_hours_start` | TIME NULL | 방해 금지 시작 시각 |
| `quiet_hours_end` | TIME NULL | 방해 금지 종료 시각 |
| `digest_mode` | TEXT NULL | `instant`, `daily`, `weekly` 등 |
| `created_at` | TIMESTAMPTZ | 생성 시각 |
| `updated_at` | TIMESTAMPTZ | 수정 시각 |

기본값 권장:

- `schedule_enabled` = true
- `study_enabled` = true
- `devotional_enabled` = true
- `prayer_enabled` = true
- `reading_weekly_enabled` = true
- `service_notice_enabled` = true
- `digest_mode` = `instant`

메모:

- 초기에 quiet hours와 digest_mode는 UI 없이 컬럼만 먼저 둬도 된다.

## 5.3 `push_deliveries`

목적:

- 어떤 알림이 어떤 구독으로 발송되었는지 추적한다.

권장 컬럼:

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `id` | UUID PK | 발송 로그 식별자 |
| `notification_id` | UUID FK -> `notifications.id` | 원본 알림 |
| `subscription_id` | UUID FK -> `push_subscriptions.id` | 대상 구독 |
| `user_id` | UUID FK -> `users.id` | 수신 사용자 |
| `district_id` | UUID FK -> `districts.id` | 수신 시점 구역 |
| `delivery_type` | TEXT | `push`, 향후 `email`, `sms` 확장 대비 |
| `status` | TEXT | `pending`, `sent`, `failed`, `expired`, `skipped` |
| `error_message` | TEXT NULL | 실패 메시지 |
| `response_code` | INTEGER NULL | 푸시 응답 코드 |
| `attempt_count` | INTEGER | 재시도 횟수 |
| `sent_at` | TIMESTAMPTZ NULL | 실제 발송 시각 |
| `created_at` | TIMESTAMPTZ | 생성 시각 |
| `updated_at` | TIMESTAMPTZ | 수정 시각 |

권장 인덱스:

- `(notification_id)`
- `(subscription_id, status)`
- `(user_id, created_at desc)`
- `(status, created_at)`

## 6. 앱 서비스 운영 공지 구조

현재 구역 단위 알림 외에 서비스 운영 공지가 필요하므로, 아래 두 방향 중 하나를 선택한다.

### 방향 A. `notifications`에 scope 추가

추가 컬럼 예시:

- `scope_type` = `district` | `service`
- `scope_target_id` = district id 또는 null
- `created_by_admin` = bool

장점:

- 한 테이블에서 관리 가능

주의:

- 기존 구역 단위 정책과 섞이지 않게 RLS와 쿼리 설계가 더 복잡해진다.

### 방향 B. 별도 `service_notifications` 테이블

장점:

- 구역 공지와 서비스 공지를 논리적으로 분리 가능
- 운영 정책이 명확함

초기 권장:

- **방향 A를 피하고, 가능하면 별도 테이블 또는 별도 scope 구조를 명확히 두는 편이 안전하다.**

## 7. 알림 타입 체계 권장

`notifications`에 아래 컬럼 추가를 검토한다.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `notification_type` | TEXT | `schedule`, `schedule_rsvp`, `study`, `devotional`, `prayer`, `reading_weekly`, `service_notice` |
| `payload` | JSONB NULL | 링크, 대상 id, 메타데이터 |

권장 이유:

- 웹푸시 문구 조합이 쉬워진다.
- 알림 설정과 타입 매칭이 쉬워진다.
- 향후 앱 내 알림 UX 분류가 가능해진다.

## 8. 권장 RLS 방향

## 8.1 `push_subscriptions`

원칙:

- 사용자는 본인 구독만 조회/등록/비활성화 가능
- 관리자 서비스 계정은 발송 처리용 전체 접근 가능

## 8.2 `notification_preferences`

원칙:

- 사용자는 본인 설정만 조회/수정 가능
- 운영 관리자 또는 서비스 역할은 발송 계산 시 조회 가능

## 8.3 `push_deliveries`

원칙:

- 일반 사용자는 본인 발송 로그 조회를 굳이 허용하지 않아도 된다.
- 운영 관리자 또는 서비스 역할만 전체 조회

## 9. 이벤트별 발송 규칙 초안

### 9.1 일정 등록

- 대상: 해당 구역 구독 사용자
- 타입: `schedule`

### 9.2 참여 여부 확인 일정

- 대상: 해당 구역 구독 사용자
- 타입: `schedule_rsvp`
- 문구 예시: `새 일정이 등록되었습니다. 참여 여부를 알려주세요.`

### 9.3 성경공부 발행

- 대상: 해당 구역 구독 사용자
- 타입: `study`

### 9.4 오늘의 묵상 게시

- 대상: 해당 구역 또는 사용자 선호 설정 기준
- 타입: `devotional`

### 9.5 신규 중보기도제목

- 대상: 해당 구역 구독 사용자
- 타입: `prayer`

### 9.6 성경읽기 주간 알림

- 대상: 해당 구역 사용자 또는 개인별 수신자
- 타입: `reading_weekly`
- 발송 시점: 매주 일요일 아침

## 10. 성경읽기 주간 알림 데이터 고려사항

이 알림은 메시지 안에 지난주 읽은 장 수가 들어간다.

따라서 발송 전 계산이 필요하다.

입력:

- 사용자별 지난주 일요일~토요일 합계

출력 예시:

- `지난주(일요일~토요일) 읽은 장수는 12장입니다. 추가할 장수가 있으면 알려주세요.`

구현 선택지:

- 발송 직전 집계 쿼리
- 별도 주간 집계 테이블 사용

초기 권장:

- **발송 직전 집계 쿼리로 시작하고, 부하가 커지면 별도 집계 테이블로 이동**

## 11. AI 확장을 위한 초안 테이블

AI 1차/2차 기능을 위해 아래 테이블도 검토할 수 있다.

### `sermon_drafts`

목적:

- 수동 입력된 설교/예배 스크립트를 AI로 정리한 초안을 저장

권장 컬럼:

- `id`
- `district_id`
- `source_type` (`sunday_sermon`, `wednesday_service`, `special_sermon`)
- `title`
- `source_text`
- `ai_summary`
- `ai_key_points`
- `ai_application_points`
- `status` (`draft`, `ready`, `published`)
- `created_by`
- `reviewed_by`
- `created_at`
- `updated_at`

### `worship_recommendations`

목적:

- 상황별 찬양 추천 결과 캐시 또는 기록

권장 컬럼:

- `id`
- `user_id`
- `context_type` (`qt`, `prayer`, `reading`)
- `context_text`
- `recommendations`
- `created_at`

메모:

- 이 테이블은 반드시 1차에 필요하지는 않다.
- 초기에는 요청 시 실시간 생성 후 저장하지 않아도 된다.

## 12. 마이그레이션 작성 순서 권장

1. `notifications` 타입 컬럼 확장 여부 결정
2. `push_subscriptions` 생성
3. `notification_preferences` 생성
4. `push_deliveries` 생성
5. RLS 정책 추가
6. 발송용 서비스 함수 또는 Edge Function 연결

## 13. 현재 결론

초기 웹푸시 구조는 아래처럼 가는 것이 가장 적절하다.

1. `notifications`를 유지한다.
2. `push_subscriptions`, `notification_preferences`, `push_deliveries`를 추가한다.
3. 성경읽기 알림은 주간 스케줄형으로 시작한다.
4. 앱 서비스 운영 공지는 구역 공지와 분리된 구조로 관리한다.
