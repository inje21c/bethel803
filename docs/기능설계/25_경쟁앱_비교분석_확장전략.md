## 1. 시장 구도: 3개 세그먼트

국내 교회 앱 시장은 기능 축으로 보면 세 덩어리다.

| 세그먼트 | 대표 서비스 | 특징 |
|----------|------------|------|
| A. 교적/행정 관리 | 온맘교적(100인 이하 무료), OKSolomon, 디모데, 오직교회관리 | 사무실용. 교적부·헌금·재정·기부금영수증. 데스크톱 출신, 관리자 중심 |
| B. 공동체/소그룹 운영 | **브랜치스**, 코이노니아, Church Round, 구함, 치윰, 만나 | 모바일 우선. 구성원·모임·소통 중심. 대부분 무료 기반 |
| C. 단일 기능 도구 | ChurchQR(QR 출석·주보), 체리(헌금/기부 결제), 스마트주보 | 한 가지 문제만 깊게. 교회는 여러 개를 조합해서 사용 |

bethel803은 **B 세그먼트**에 속하되, A·C의 일부 기능(출석, 주보, 통계)을 소그룹 범위에서 흡수한 형태다.

## 2. 주요 경쟁 서비스 분석

### 브랜치스 (branches.team) — 가장 직접적인 경쟁자
- **타깃**: 교회 청년부·청소년부·교육부 (사역자/임원/교사)
- **기능**: 구성원·셀 관리, 모바일 주보 제작, 행사/일정, 재정(영수증+구글시트 연동), QT 묵상, 기도제목 공유, 셀 모임, 예배 기록
- **실적**: 전국 415개 공동체, 3,547명 사용 (2026 기준 홈페이지)
- **가격**: 무료 기반 (명시된 유료 플랜 없음)
- **평가**: 기능 집합이 bethel803과 가장 많이 겹친다(QT/기도/셀/일정). 다만 **청년·교육부서에 특화**되어 있고, 공동체당 평균 사용자 수(약 8.5명)로 보아 소규모 부서 단위 채택이 많다.

### 코이노니아 (koinonia.it.kr)
- **타깃**: 교회/본당 전체. 전례력·세례명 프로필 등 **가톨릭 색채**가 강함
- **기능**: 신자(교적) 관리, 분과/단체 관리, 실시간 채팅, 공지·경조사 알림, 자료 영구 보관
- **차별점**: 관리자 승인 기반 폐쇄형 커뮤니티, 채팅 파일 영구 보존(카톡 대비 강점 소구)
- **평가**: "교회 전용 카톡+교적". 개신교 구역 모임의 신앙 콘텐츠(성경읽기/QT/말씀공부)는 없음

### Church Round (App Store, Surfmind)
- **타깃**: 교회 전체 (관리자 웹 + 신도 앱)
- **기능**: 교인 요람, 교회 소식, 커뮤니티(물품 나눔, 사역자 모집)
- **평가**: 2025년 출시 신생 서비스, 무료. 요람/소식 중심이라 소그룹 운영 도구는 아님

### ChurchQR (churchqr.net)
- **타깃**: 출석 관리가 필요한 교회 (코로나 전자출입명부 출발, 태국 선교사 개발)
- **기능**: QR 출석 체크, 실시간 출결 통계·엑셀 출력, 온라인 주보
- **평가**: 출석이라는 단일 문제를 깊게 해결. 소그룹 신앙 콘텐츠 없음

### 참고: 인접 서비스
- **체리(cherrycorp.io)**: 블록체인 기반 헌금/기부 결제 플랫폼 + QR 체크인. 수익모델이 '결제'라서 운영 도구와는 다른 궤도. 헌금 기능은 우리가 만들 영역이 아니라 **제휴 후보**
- **구함**: 크리스천 연합 플랫폼(채용·공간·장터·채팅). 교회 간 네트워크 지향, 단일 교회 운영 도구 아님
- **만나/치윰**: 교적+커뮤니티 통합형 무료 솔루션. A↔B 경계 침식 중

## 4. 포지셔닝 결론

### 4.1 우리의 빈자리(니치)는 명확하다

경쟁 서비스는 모두 **"운영(사람·일정·소통) 관리"**에 머물러 있고, 브랜치스만 신앙 콘텐츠(QT/기도)를 일부 다루나 **청년부 특화**다. bethel803의 차별점은:

1. **장년 구역(속회/목장/구역) 시장** — 브랜치스가 비워둔 자리. 한국 교회의 소그룹 주류는 청년부가 아니라 장년 구역 조직이다
2. **신앙 콘텐츠 자동화 (AI)** — 묵상 자동 수집·요약, 주보 PDF→성경공부 교재 자동 생성, 주간 보고 자동 마감. 어떤 경쟁사도 없음. 구역장의 "매주 콘텐츠 준비 노동"을 없애는 것이 핵심 가치
3. **성경읽기 깊이** — 구조화 리더 + 읽기 플랜 + 구역 단위 통계는 독자 영역

→ 포지셔닝 문장: **"구역장이 아무것도 준비하지 않아도 매주 구역 모임이 돌아가게 하는 앱"**

### 4.2 싸우지 않을 영역 (의도적 포기)

- **채팅**: 카톡과 경쟁 불가. 코이노니아의 길은 가지 않는다. 알림은 웹푸시+카톡 공유로 충분
- **헌금/재정 결제**: 결제업·전자금융 규제 영역. 필요 시 체리류와 제휴
- **전교회 교적부**: A 세그먼트 기존 강자(온맘 등)와 정면 경쟁 회피. 우리는 "구역 명부"까지만
- **주보 제작**: 브랜치스가 잘함. 우리는 주보를 "소비(파싱)"하는 쪽

### 4.3 경계해야 할 리스크

- **브랜치스의 장년 확장**: 기능 겹침이 가장 크다. 그들이 구역 시장으로 내려오기 전에 장년 구역 레퍼런스를 먼저 쌓아야 한다
- **전부 무료인 시장**: B 세그먼트 가격은 사실상 0원. 유료 전환 여지는 'AI 기능 원가'에만 있음 (23번 문서의 하이브리드 모델과 일치 — 전략 유지)
- **개인정보(민감정보) 신뢰**: 교인 데이터는 종교 민감정보. 신생 서비스가 신뢰를 얻는 길은 "데이터 소유권은 교회에"(언제든 엑스포트/반환) 명시 — 경쟁사 대부분이 약한 부분

## 5. 실행 전략

### 5.1 확장 전 제품 보강 (경쟁 비교에서 드러난 갭)

| 우선순위 | 항목 | 근거 |
|---|------|------|
| 1 | **교회별 콘텐츠 소스 설정** (주보 URL, 묵상 소스) | AI 자동화가 차별점인데 현재 벧엘 하드코딩 → 타교회에서 가치 제로 |
| 2 | 데이터 엑스포트 강화 (구역 단위 전체 백업) | "데이터는 교회 소유" 신뢰 전략의 실체 |
| 3 | QR 출석 (선택 기능) | ChurchQR 수요 흡수. 일정-출석 구조가 이미 있어 추가 비용 작음 |
| 후순위 | 모바일 주보 제작, 재정, 채팅 | 4.2 의도적 포기 또는 보류 |

### 5.2 Go-to-market (23번 문서 Phase 9~12에 덧붙임)

1. **레퍼런스 우선**: 벧엘 + 지인 네트워크 교회 2~3곳 파일럿 (무료, 수동 온보딩) — 브랜치스도 415개를 입소문으로 만들었다
2. **타깃 메시지**: 담임목사가 아니라 **구역장/교구 담당 교역자**에게. "구역 모임 준비 시간 주 2시간 → 0"
3. **채택 단위**: 교회 전체가 아니라 **교구/구역 단위 부분 도입 허용** (브랜치스의 부서 단위 채택 패턴 검증됨) — 다중 구역 구조가 이미 이를 지원
4. **가격**: 1단계 전액 무료 + 후원, AI 사용량이 늘면 Free/Standard 분기 (23번 문서 그대로, 이번 분석으로 재확인: 시장가 0원이므로 기본 기능 유료화는 불가)
5. **신뢰 자산**: 개인정보 처리방침 + 데이터 반환 절차를 온보딩 문서에 명시

### 5.3 즉시 액션

1. 23번 문서 Phase 10에 "QR 출석(선택)"과 "데이터 엑스포트 강화" 반영
2. church_settings 설계 시 콘텐츠 소스(주보 URL 패턴, 묵상 소스, 파싱 스케줄)를 1급 설정으로
3. 파일럿 후보 교회 리스트업 (지인 네트워크) — 코드보다 먼저 시작 가능
4. 랜딩 문구 초안: "구역 모임, 준비 없이 모이세요" — 묵상/교재/보고가 자동으로 준비되는 데모 영상 1개

## 6. 브랜치스 온보딩 분석과 범용화 설계 원칙 (2026-06-11 추가)

브랜치스 실제 가입 흐름 캡처 분석: 소셜 로그인(카카오/구글 우선) → 약관 동의 →
프로필(실명/전화/성별/생년월일) → 공동체 검색·선택 또는 **"가입 후 직접 생성하기"** → 홈.

### 6.1 핵심 교훈: 632개 공동체, SI 0건

브랜치스의 "개별 모임별 설정"의 실체는 커스터마이징이 아니라:

1. **계정-공동체 분리**: 가입은 개인이 하고, 공동체는 검색해 참여하거나 셀프서비스로 생성
2. **기능 모듈화**: 홈의 아이콘 그리드(설문/셀모임/재정/회의록/기도제목/찬양콘티)가
   모듈 단위이고, 공동체가 골라 쓴다
3. 약관/개인정보 동의가 가입 플로우의 정식 단계

### 6.2 "bethel803은 너무 특화됐다"는 진단의 분해

| 기능 | 벧엘 전용? | 범용화 방법 |
|------|---|---|
| 묵상 자동 수집 | 아니다 (성서유니온 매일성경 = 한국 교회 표준 교재) | 그대로 범용, 공통 콘텐츠 강점 |
| QT/성경읽기/기도/일정 | 아니다 | 그대로 |
| "구역" 용어 | 명칭만 (속회/목장/셀/순) | 용어 사전 설정 |
| 주간 보고 양식 | 부분적 | 템플릿화 또는 모듈 off |
| 주보 PDF 파싱 | **그렇다** (URL 패턴·양식) | 프롬프트 일반화 + 교회별 URL 설정, 안 맞으면 모듈 off |

→ 결론: SI 커스터마이징은 하지 않는다. 범용화는 **3단 흡수**로 해결한다:
① 설정으로 흡수(용어/소스/마감 요일) ② 모듈 on/off로 흡수 ③ 컨벤션(기본값)으로 흡수
(설정 없이도 동작: 매일성경 묵상 기본 ON, 주보 파싱 기본 OFF)

### 6.3 지킬 것

브랜치스는 빈 그릇(공동체가 콘텐츠를 채움), 우리는 채워진 그릇(콘텐츠 자동 준비).
범용화 대상은 설정과 껍데기이지 자동화 엔진이 아니다.

### 6.4 Phase 10 반영 사항

1. church_settings = 용어 사전 + 콘텐츠 소스 + **기능 모듈 플래그**
2. 가입 플로우에 약관/개인정보 수집 동의 단계 추가 (민감정보 — 법적 필수)
3. 셀프서비스 교회 생성("직접 생성하기")은 교회 수 두 자릿수 시점으로 후순위
4. parse-bulletin 프롬프트의 벧엘 의존도 점검 → 범용 파서 전환
