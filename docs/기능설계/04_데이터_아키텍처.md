# Data Architecture / 데이터 아키텍처

> Public document / 공개 문서
> Last reviewed / 마지막 점검: 2026-07-07

## English Summary

bethel803 uses a Supabase-centered architecture. The React frontend talks to Supabase through a typed access layer, while PostgreSQL and Row Level Security enforce the final data boundaries. The key architectural principle is group-scoped care: leaders operate within their own group, and higher-level administration is separated from everyday member workflows.

## 한국어 요약

bethel803은 Supabase 중심 구조를 사용합니다. React 프론트엔드는 타입이 정리된 접근 계층을 통해 Supabase와 통신하고, 최종 데이터 경계는 PostgreSQL과 RLS가 강제합니다. 핵심 원칙은 모임 단위 돌봄입니다. 리더는 자기 모임 범위 안에서 운영하고, 상위 관리 기능은 일반 구성원의 일상 흐름과 분리합니다.

## 2. Storage Model / 저장 구조

### 2.1 Browser Storage / 브라우저 저장소

Core business data is not stored in `localStorage`. Browser storage is limited to session support and small UI preferences.

핵심 업무 데이터는 `localStorage`에 저장하지 않습니다. 브라우저 저장소는 세션 유지와 작은 UI 상태 저장에 제한적으로 사용합니다.

| Key | Purpose |
| --- | --- |
| Supabase auth storage | Session persistence / 세션 유지 |
| `bethel-popup-{date}` | Schedule popup state / 일정 팝업 표시 상태 |
| `bethel-debug` | Debug logging flag / 디버그 로그 플래그 |

### 2.2 Server Source of Truth / 서버 기준 진실 공급원

The source of truth is Supabase: Auth, PostgreSQL, RLS, Storage, and Edge Functions.
단일 진실 공급원은 Supabase Auth, PostgreSQL, RLS, Storage, Edge Functions입니다.

## 4. Application Access Layer / 애플리케이션 접근 계층

The frontend does not use a separate REST API server for the main product flows. It accesses Supabase through shared libraries.

프론트엔드는 주요 제품 흐름에서 별도 REST 서버를 두지 않고 공통 라이브러리를 통해 Supabase에 접근합니다.

| File | Responsibility |
| --- | --- |
| `src/lib/supabase.ts` | Browser Supabase client / 브라우저용 Supabase 클라이언트 |
| `src/lib/authContext.tsx` | Session, profile, role, status / 세션, 프로필, 권한, 상태 |
| `src/lib/api.ts` | Feature data access and type conversion / 기능별 데이터 접근과 타입 변환 |
| `src/lib/churchContext.tsx` | Church and plan context / 교회와 플랜 맥락 |
| `src/lib/districtContext.tsx` | Active group/district context / 활성 모임·구역 맥락 |

Design rules / 설계 원칙:

- Pages should prefer shared access functions over ad hoc queries.
  페이지는 임의 쿼리보다 공통 접근 함수를 우선 사용합니다.
- Database column names and UI property names are translated at the access layer.
  DB 컬럼명과 UI 속성명 차이는 접근 계층에서 변환합니다.
- Final authorization belongs to RLS, not only to UI checks.
  최종 권한 보호는 UI가 아니라 RLS가 담당합니다.

## 5. Representative Data Flows / 대표 데이터 흐름

### Authentication / 인증

```text
User input / 사용자 입력
 -> Supabase Auth
 -> users profile lookup / users 프로필 조회
 -> role and status check / 권한과 상태 확인
 -> route decision / 라우트 분기
```

### Bible Study / 성경공부

```text
BibleStudyList or Detail
 -> api.ts
 -> bible_studies / study_answers
 -> answer save and completion state
```

### Prayer Requests / 기도제목

```text
PrayerRequests or Detail
 -> api.ts
 -> prayer_requests / prayer_responses / prayer_intercessions
 -> sharing scope and intercession count
```

### Schedule & Attendance / 일정과 출석

```text
ScheduleManagement
 -> api.ts
 -> schedules / attendances
 -> optional Storage attachment
```

### Automation / 자동화

```text
Scheduled job or manual trigger
 -> Edge Function or database function
 -> external source or AI processing when configured
 -> product tables
```

## 6. Authorization Model / 권한 구조

| Field | Values |
| --- | --- |
| `role` | `master`, `leader`, `member` |
| `status` | `pending`, `active` |

Rules / 원칙:

- `pending` users are directed to the approval waiting screen.
  `pending` 사용자는 승인 대기 화면으로 이동합니다.
- `leader` users operate within their own group scope.
  `leader`는 자기 모임 범위 안에서 운영합니다.
- `master` users handle church-level administration.
  `master`는 교회 단위 관리를 담당합니다.
- RLS enforces data isolation by `church_id` and `district_id`.
  RLS가 `church_id`, `district_id` 기준 데이터 격리를 강제합니다.
