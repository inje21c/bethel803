## 2. 현재 저장 구조

### 2.1 프론트 로컬 상태

현재 앱의 핵심 업무 데이터는 `localStorage`가 아니라 Supabase를 기준으로 동작한다.

브라우저 저장소는 주로 다음 용도로 사용된다.

| 키 | 내용 |
|----|------|
| Supabase auth storage | 세션 유지 |
| `bethel-popup-{date}` | 일정 팝업 표시 여부 |
| `bethel-debug` | 디버그 로그 활성화 |

### 2.2 서버 기준 저장소

현재 기준 단일 진실 공급원은 `Supabase(Auth + PostgreSQL + Storage + Edge Functions)`다.

## 4. 애플리케이션 접근 구조

현재 프론트는 REST 서버를 별도로 두지 않고 Supabase SDK를 통해 직접 접근한다.

핵심 계층:

- `src/lib/authContext.tsx`: 세션 감시, 프로필 로딩, 권한 판단
- `src/lib/api.ts`: 기능별 데이터 접근, 타입 변환, timeout 처리
- `src/lib/supabase.ts`: 브라우저용 Supabase 클라이언트

원칙:

- 페이지는 가능하면 직접 쿼리하지 않고 `api.ts`를 통해 접근한다.
- DB 컬럼명과 UI 속성명 차이는 접근 계층에서 변환한다.
- 최종 권한 보호는 화면이 아니라 RLS가 담당한다.

## 6. 권한 구조

현재 권한 체계는 `users.role` 및 `users.status` 기준으로 동작한다.

| 필드 | 값 |
|------|----|
| `role` | `master`, `leader`, `member` |
| `status` | `pending`, `active` |

권한 원칙:

- `pending` 사용자는 승인 대기 화면으로 이동한다.
- `leader`는 자기 구역 범위의 운영 기능을 수행한다.
- `master`는 전체 구역 관리와 구역 전환이 가능하다.
- 데이터 보호는 RLS 정책으로 강제한다.

## 7. 자동화 스케줄 기준

운영 목표 스케줄은 다음과 같다.

| 시각 | 작업 |
|------|------|
| 매일 06:00 KST | 오늘의 묵상 수집 및 요약 |
| 매주 일요일 오전 | 주간 마감 + 보고서 생성 |
| 매주 일요일 20:00 KST | 주보 PDF 수집 + 성경공부 초안 생성 |

주의:

- 저장소 기준 스케줄 SQL 예시는 존재한다.
- 실제 운영 등록 여부는 별도 점검이 필요하다.

## 8. 기술 부채 및 남은 항목

| 항목 | 현황 | 다음 방향 |
|------|------|----------|
| 운영 환경 점검 | 코드 구현됨 | cron, Function, Storage 실배포 확인 |
| 문서 최신화 | 진행 중 | 현재 구조 기준으로 정리 |
| PWA | 의존성만 존재 | 실제 적용 여부 결정 및 구현 |
| 성능 | 빌드 통과 | 관리자 화면 청크 최적화 |
| 폼 검증 | 기본 수준 | 필요 시 RHF/Zod 확장 |
| 접근성 | 기본 수준 | 주요 폼/다이얼로그 추가 점검 |

## 9. 구현 판단 기준

- 사용자, 권한, 동기화가 필요한 데이터는 모두 서버 기준으로 본다.
- 운영 자동화는 코드 구현과 운영 활성화를 분리해서 판단한다.
- 문서 내 데이터 모델이나 권한 구조가 바뀌면 이 문서를 먼저 갱신한다.
- 운영 시작 전 최종 점검은 [운영준비체크리스트.md](/home/ubuntu/bethel803/docs/운영준비체크리스트.md)를 기준으로 한다.
