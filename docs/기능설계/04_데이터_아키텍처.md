# 데이터 및 아키텍처

## 1. 라우트 맵

```text
/                      로그인
/dashboard             대시보드
/bible-study           성경 공부 목록
/bible-study/:id       성경 공부 상세
/prayer-requests       기도 제목
/bible-reading         성경 읽기
/schedule              일정 관리
/admin                 관리자 대시보드
```

추가 후보 라우트:

```text
/admin/members         구역원 관리
/notifications         알림 센터
/profile               내 프로필
```

## 2. 현재 저장 구조

| 키 | 내용 |
|----|------|
| `bethel-user` | 로그인 사용자 |
| `bethel-answers` | 성경 공부 답변 |
| `bethel-prayers` | 기도 제목 |
| `bethel-readings` | 성경 읽기 기록 |
| `bethel-schedules` | 일정 목록 |
| `bethel-attendances` | 출석 기록 |
| `bethel-popup-{date}` | 일정 팝업 표시 여부 |

## 3. 백엔드 전환 방향

권장 구조는 `Supabase(Auth + PostgreSQL + Storage + Edge Functions)` 기준으로 정리한다.

### 3.1 핵심 테이블

| localStorage | 목표 테이블 |
|-------------|------------|
| `bethel-user` | `users` |
| `bethel-answers` | `study_answers` |
| `bethel-prayers` | `prayer_requests` |
| `bethel-readings` | `bible_reading_logs` |
| `bethel-schedules` | `schedules` |
| `bethel-attendances` | `attendances` |
| 신규 | `bible_studies`, `daily_devotionals`, `weekly_reports`, `notifications` |

### 3.2 필수 백엔드 기능

- 실제 회원가입/로그인/비밀번호 재설정
- 회원가입 승인 흐름
- 사용자별 데이터 영속 저장
- RLS 기반 권한 분리
- 자동화 작업용 CRON 및 Edge Function

## 4. 대표 API 범위

### 4.1 인증

```text
POST /api/auth/register
POST /api/auth/login
POST /api/auth/logout
POST /api/auth/reset
PATCH /api/auth/approve/:id
```

### 4.2 성경 공부

```text
GET    /api/studies
GET    /api/studies/:id
POST   /api/studies
PATCH  /api/studies/:id
DELETE /api/studies/:id
POST   /api/studies/scrape
PATCH  /api/studies/:id/publish
PATCH  /api/studies/:id/close
```

### 4.3 묵상 및 보고서

```text
GET  /api/devotional/today
GET  /api/devotional/:date
POST /api/devotional/scrape

GET  /api/reports/weekly
GET  /api/reports/weekly/:weekNum
POST /api/reports/weekly/generate
```

### 4.4 운영 부가 기능

```text
GET    /api/members
PATCH  /api/members/:id
DELETE /api/members/:id

GET    /api/notifications
PATCH  /api/notifications/:id
DELETE /api/notifications/:id
```

## 5. 자동화 스케줄

| 시각 | 작업 |
|------|------|
| 매일 06:00 KST | 오늘의 묵상 수집 및 요약 |
| 매일 06:30 KST | 묵상 재시도 |
| 매주 일요일 11:20 KST | 주간 마감 + 보고서 생성 |
| 매주 일요일 20:00 KST | 주보 PDF 수집 + 성경공부 생성 |

## 6. 기술 부채

| 항목 | 현황 | 개선 방향 |
|------|------|----------|
| 테스트 | placeholder 수준 | Vitest + RTL |
| 에러 처리 | 전역 경계 없음 | Error Boundary 추가 |
| 폼 검증 | 일부만 적용 | React Hook Form + Zod 확장 |
| 역할 판단 | 이름 문자열 기반 | 서버 role 필드 사용 |
| 하드코딩 데이터 | 성경 공부 3건 코드 내 포함 | DB 동적 조회 |
| 접근성 | aria 보강 필요 | 주요 입력/버튼 보완 |

## 7. 구현 판단 기준

- 사용자/권한/동기화가 필요하면 무조건 서버 기준으로 설계한다.
- 운영 자동화는 수동 기능이 먼저 안정화된 뒤 붙인다.
- 문서 내 데이터 모델 변경 시 이 문서를 먼저 수정한다.
