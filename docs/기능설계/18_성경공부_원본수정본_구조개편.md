# 성경공부 원본/수정본 구조 개편

## 1. 문서 목적

이 문서는 주보 PDF 파싱 기반 성경공부 등록 구조를 `구역별 자동 초안 생성` 방식에서 `원본 1건 + 구역별 수정본` 방식으로 개편하기 위한 기준 문서다.

핵심 목표는 다음과 같다.

- 같은 일자의 주보 파싱 원본은 시스템 전체에서 1건만 유지한다.
- 구역별 수정과 발행은 기존 `bible_studies`에서 계속 수행한다.
- 구역원은 자기 구역에서 발행한 성경공부만 본다.
- 구역장은 모든 원본을 볼 수 있지만, 수정본은 자기 구역 것만 편집한다.

## 2. 현재 문제

현재 [parse-bulletin](/home/ubuntu/bethel803/supabase/functions/parse-bulletin/index.ts)는 파싱 결과를 활성 구역 수만큼 [bible_studies](/home/ubuntu/bethel803/src/lib/api.ts)에 직접 생성한다.

이 방식의 문제는 다음과 같다.

- 같은 주차 자료가 구역 수만큼 중복 생성된다.
- 원본과 구역별 수정본의 역할이 한 테이블 안에서 섞인다.
- 어떤 row가 원본인지, 어떤 row가 발행본인지 구분이 어렵다.
- 다른 구역장이 같은 날짜 자료를 다시 파싱하면 데이터가 더 중복된다.

## 3. 목표 동작

### 3.1 파싱 원본

- 자동 파싱이든 수동 파싱이든 같은 `study_date`에 대한 원본은 1건만 허용한다.
- 이미 동일 `study_date`의 원본이 있으면 새 파싱은 수행하지 않는다.
- 이 경우 함수는 경고 메시지와 함께 `409 Conflict`를 반환한다.
- 원본은 모든 구역장이 조회할 수 있다.

### 3.2 구역별 수정본

- 구역별 수정본은 기존 `bible_studies`에 저장한다.
- 수정본은 각 구역이 원본을 바탕으로 복제해 만든다.
- 각 구역은 자기 구역의 수정본만 수정/발행할 수 있다.
- 같은 구역에서 같은 원본에 대한 수정본은 1건만 허용한다.

### 3.3 구역원 노출

- 구역원은 자기 구역의 `published = true` 성경공부만 본다.
- 원본은 구역원에게 직접 노출하지 않는다.

## 4. 권장 데이터 구조

### 4.1 신규 테이블: `study_sources`

`study_sources`는 주보 파싱 원본을 저장한다.

권장 컬럼:

- `id`
- `study_date`
- `week_number`
- `title`
- `scripture`
- `introduction`
- `questions`
- `source_pdf_url`
- `parse_mode`
- `parsed_by`
- `created_at`
- `updated_at`

제약:

- `study_date` unique
- `parse_mode in ('auto', 'manual')`

### 4.2 기존 테이블: `bible_studies`

`bible_studies`는 구역별 수정본과 발행본을 담당한다.

추가 권장 컬럼:

- `source_id UUID NULL REFERENCES study_sources(id) ON DELETE SET NULL`
- `source_snapshot JSONB NOT NULL DEFAULT '{}'::jsonb`

제약:

- 같은 구역에서 같은 원본을 중복 생성하지 않도록 `UNIQUE (district_id, source_id)`
- `source_id`가 없는 기존 수동 등록 row는 허용

## 5. 파싱 함수 동작 변경

대상 함수:

- [parse-bulletin](/home/ubuntu/bethel803/supabase/functions/parse-bulletin/index.ts)

새 동작:

1. PDF URL에서 날짜를 추론하거나 GPT 결과에서 `study_date`를 산출한다.
2. `study_sources`에서 같은 `study_date` 존재 여부를 확인한다.
3. 이미 있으면 `409`와 기존 원본 id/title/date를 반환한다.
4. 없으면 `study_sources`에 원본 1건만 저장한다.
5. 더 이상 구역별 [bible_studies](/home/ubuntu/bethel803/src/lib/api.ts)에 자동 생성하지 않는다.

권장 응답 형식:

```json
{
  "ok": true,
  "sourceId": "uuid",
  "studyDate": "2026-04-05",
  "title": "영생을 가졌는가?",
  "created": true
}
```

중복 시 권장 응답 형식:

```json
{
  "ok": false,
  "code": "SOURCE_ALREADY_EXISTS",
  "sourceId": "uuid",
  "studyDate": "2026-04-05",
  "title": "영생을 가졌는가?"
}
```

## 6. 관리자/구역장 UX 변경

### 6.1 관리자/구역장 공통

- 원본 목록 조회
- PDF 출처 링크 확인
- 원본 내용 미리보기

### 6.2 구역장

- 원본에서 `내 구역 수정본 만들기`
- 이미 수정본이 있으면 `수정본 편집하기`
- 수정본 저장 후 `published=true`로 발행

### 6.3 경고 UX

- 같은 날짜 원본이 이미 있을 경우:
  - "이미 2026-04-05 원본이 등록되어 있습니다. 새 파싱은 수행하지 않았습니다."
- 다른 구역장이 다시 수동 파싱을 시도해도 동일한 경고를 보여준다.

## 8. 데이터 마이그레이션 전략

### 8.1 신규 원본만 적용하는 최소 전략

- 기존 `bible_studies`는 그대로 둔다.
- 새 파싱부터만 `study_sources`를 사용한다.
- 과거 row는 `source_id = NULL`로 유지한다.

이 방식이 가장 안전하다.

### 8.2 과거 데이터 연결은 후속 작업

- 이미 발행된 과거 성경공부를 역으로 원본에 연결하는 작업은 선택 사항이다.
- 운영 절체 직후에는 필수 아님.

## 9. 구현 순서 권장

1. `study_sources` 테이블 추가
2. `bible_studies.source_id`, `source_snapshot` 추가
3. `parse-bulletin`을 원본 1건 생성 방식으로 변경
4. 관리자 화면에 원본 목록 추가
5. `내 구역 수정본 만들기` 버튼 추가
6. 구역별 수정본 중복 방지
7. 발행 흐름 점검

## 10. 운영 판단

이 구조는 다음 요구를 가장 잘 만족한다.

- 주보 파싱 원본은 시스템 전체에서 중복 없이 관리
- 구역별 수정 자유도 보장
- 구역원 노출 범위 명확화
- 자동파싱과 수동파싱의 충돌 제거

즉, 성경공부 자료는 `원본 공유 + 구역별 발행`의 2단 구조로 가는 것이 맞다.
