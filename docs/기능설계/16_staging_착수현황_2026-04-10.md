## 5. 중요한 현재 리스크

### 5.1 운영 배포는 아직 Firebase 기준

이 저장소의 현재 운영 기준선은 아직 Firebase다. 따라서 staging 준비 중이라도 운영 긴급 수정은 기존 배포 흐름을 따라가야 한다.

### 5.2 EC2 개발 `.env.local`이 아직 운영 기준선을 보고 있음

현재 EC2의 [`.env.local`](/home/ubuntu/bethel803/.env.local) 은 운영 URL 성격의 값으로 보인다.

확인 명령:

```bash
npm run staging:env-check
```

의미:

- staging Supabase가 생기기 전까지는 개발 환경이 운영 기준선에 가까운 값을 보고 있다.
- staging 프로젝트를 만든 뒤 `.env.local`을 staging 값으로 바꾸는 순간이 실제 전환의 첫 번째 안전장치다.

### 5.3 Supabase 로컬 프로젝트 설정은 아직 없음

현재 저장소에는 `supabase/config.toml`이 없다. 즉, 이 저장소는 Supabase 로컬 개발 워크플로까지 정리된 상태는 아니고, 마이그레이션과 함수 초안을 저장하는 구조에 가깝다.

### 5.4 staging은 콘솔 작업 없이는 시작되지 않음

아래 정보가 확보되기 전에는 [13_staging_준비체크리스트.md](/home/ubuntu/bethel803/docs/기능설계/13_staging_준비체크리스트.md)의 4.1 단계가 완료되지 않는다.

- staging Supabase 식별 정보
- Vercel project 식별 정보
- GitHub Environment 접근 권한

## 6. 지금부터의 실제 시작 순서

1. staging Supabase 프로젝트를 먼저 만든다.
2. [14_env_값입력표.md](/home/ubuntu/bethel803/docs/기능설계/14_env_값입력표.md)에 비밀이 아닌 식별 정보부터 채운다.
3. EC2 `.env.local`을 staging 값으로 전환하고 `npm run staging:env-check`를 다시 실행한다.
4. Vercel project를 연결하고 Preview env 자리를 만든다.
5. GitHub `staging` environment를 만든다.
6. `015` 적용, 함수 secret 입력, 함수 배포는 그 다음에 한다.

## 7. 로컬 반복 점검 명령

아래 명령으로 현재 준비 상태를 다시 점검할 수 있다.

```bash
bash scripts/staging_preflight.sh
npm run staging:env-check
```

이 명령은 다음을 확인한다.

- CLI 가용성
- workflow 파일 존재 여부
- Phase 8 함수/마이그레이션 존재 여부
- `.env.local` 존재 여부
- `.env.local`이 staging 성격의 값인지
- `supabase/config.toml` 존재 여부
- 현재 디스크 여유

## 8. 현재 판정

현재 상태는 `staging 착수 가능, 외부 콘솔 작업 대기`다.

즉, 로컬 기준선과 병행 운영 구조는 준비되었지만, 실제 staging은 아직 Supabase/Vercel/GitHub 콘솔 설정이 시작되지 않은 상태다.
