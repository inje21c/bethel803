# 사용자 문의 기능 상세 설계

> 요구사항 출처: 2026-06-14 세션 ("bethel803 project review and roadmap")
> 작성일: 2026-06-15 (회사)

---

## 1. 목표

앱 내에서 사용자가 문의/버그/기능 요청을 제출하면 **자동으로 GitHub 이슈가 생성**되고,
개발자가 이슈를 처리·답변하면 **사용자에게 앱 내 알림과 웹푸시**로 결과가 전달된다.
이슈 히스토리는 서비스 고도화 백로그로 누적된다.

---

## 2. 전체 흐름

```
[사용자]                [Supabase]                [GitHub]
   │                        │                        │
   │  문의 제출 (앱 폼)      │                        │
   │──────────────────────▶│                        │
   │                        │  support_tickets INSERT │
   │                        │──────────────────────▶│
   │                        │  Edge Fn: submit-ticket │
   │                        │  GitHub Issue 생성      │
   │                        │◀──────────────────────│
   │                        │  issue_number 저장     │
   │◀──────────────────────│                        │
   │  "접수 완료" 화면       │                        │
   │                        │                        │
   │              [개발자가 GitHub에서 작업]           │
   │                        │  이슈 comment/close    │
   │                        │◀──── webhook ─────────│
   │                        │  Edge Fn: github-webhook│
   │                        │  tickets 상태·답변 업데이트
   │                        │  push-dispatch 호출    │
   │◀──────────────────────│                        │
   │  웹푸시 + 앱 내 뱃지    │                        │
```

## 3. DB 설계 (마이그레이션 035)

### `support_tickets` 테이블

```sql
CREATE TABLE public.support_tickets (
  id               UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  church_id        UUID        NOT NULL REFERENCES public.churches(id) ON DELETE CASCADE,
  user_id          UUID        NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,

  -- 문의 내용
  ticket_type      TEXT        NOT NULL CHECK (ticket_type IN ('bug','feature','question','other')),
  title            TEXT        NOT NULL,
  content          TEXT        NOT NULL,

  -- 처리 상태
  status           TEXT        NOT NULL DEFAULT 'open'
                               CHECK (status IN ('open','in_progress','resolved','closed')),

  -- GitHub 연동
  github_issue_number  INTEGER,
  github_issue_url     TEXT,

  -- 답변
  admin_reply      TEXT,
  replied_at       TIMESTAMPTZ,

  created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 인덱스
CREATE INDEX ON public.support_tickets (church_id, created_at DESC);
CREATE INDEX ON public.support_tickets (user_id, created_at DESC);
CREATE INDEX ON public.support_tickets (github_issue_number) WHERE github_issue_number IS NOT NULL;
```

### RLS 정책

| 작업 | 허용 대상 | 조건 |
|------|----------|------|
| SELECT | member | 본인 티켓만 (`user_id = auth.uid()`) |
| SELECT | master | 자기 교회 전체 티켓 |
| INSERT | active member | `church_id = get_my_church_id()`, `user_id = auth.uid()` |
| UPDATE | 없음 (service role 전용) | 상태·답변은 Edge Fn에서만 갱신 |

```sql
ALTER TABLE public.support_tickets ENABLE ROW LEVEL SECURITY;

CREATE POLICY "member sees own tickets"
  ON public.support_tickets FOR SELECT
  USING (user_id = auth.uid());

CREATE POLICY "master sees church tickets"
  ON public.support_tickets FOR SELECT
  USING (
    church_id = public.get_my_church_id()
    AND EXISTS (
      SELECT 1 FROM public.users
      WHERE id = auth.uid() AND role = 'master' AND status = 'active'
    )
  );

CREATE POLICY "active member can submit"
  ON public.support_tickets FOR INSERT
  WITH CHECK (
    church_id = public.get_my_church_id()
    AND user_id = auth.uid()
    AND EXISTS (
      SELECT 1 FROM public.users
      WHERE id = auth.uid() AND status = 'active'
    )
  );
```

### updated_at 트리거

```sql
CREATE TRIGGER support_tickets_updated_at
  BEFORE UPDATE ON public.support_tickets
  FOR EACH ROW EXECUTE FUNCTION public.update_updated_at_column();
```

## 4. Edge Function 설계

### 4-1. `submit-ticket`

**트리거**: 앱에서 `supabase.functions.invoke('submit-ticket', { body: {...} })`

**처리 순서**:
1. JWT 검증 → `auth.uid()`, `church_id` 추출
2. `support_tickets` INSERT (service role)
3. GitHub REST API로 이슈 생성
4. `github_issue_number`, `github_issue_url` 업데이트
5. 성공 응답 반환

**GitHub 이슈 포맷**:
```
제목: [문의] {title} — {church_handle}

본문:
## 문의 내용
{content}

---
**유형**: {ticket_type}
**교회**: {church_name} ({church_handle})
**역할**: {user_role}
**제출일**: {created_at KST}
**티켓 ID**: {ticket_id}
```

**GitHub Labels**:
- `support` (공통)
- `bug` / `feature` / `question` / `other` (유형별)
- `church:{handle}` (교회 식별, 자동 생성)

**필요 환경변수** (Supabase Secrets):
```
GITHUB_TOKEN=ghp_...       # fine-grained PAT, issues:write
GITHUB_REPO_OWNER=inje21c
GITHUB_REPO_NAME=bethel803
```

**응답**:
```json
{ "ok": true, "ticketId": "...", "issueUrl": "..." }
```

---

### 4-2. `github-webhook`

**트리거**: GitHub Webhook → `POST /functions/v1/github-webhook`

**수신 이벤트**:
| GitHub 이벤트 | action | 처리 |
|--------------|--------|------|
| `issue_comment` | `created` | 답변 저장, status → `in_progress`, 푸시 발송 |
| `issues` | `closed` | status → `resolved`, 푸시 발송 |
| `issues` | `reopened` | status → `open` |

**처리 순서**:
1. `X-Hub-Signature-256` 검증 (`GITHUB_WEBHOOK_SECRET`)
2. issue_number로 `support_tickets` 조회
3. 이벤트별 분기 처리:
   - comment 생성: 봇/본인 코멘트 필터 후 `admin_reply` 업데이트
   - issue closed: status `resolved`
4. 해당 user_id에게 푸시 알림 발송 (`push-dispatch` 호출)
5. 204 반환

**필요 환경변수**:
```
GITHUB_WEBHOOK_SECRET=...
```

**푸시 알림 메시지**:
- 답변 수신: `"문의에 답변이 등록되었습니다"`
- 처리 완료: `"문의가 처리 완료되었습니다"`

## 8. 보안 고려사항

- GitHub PAT는 Supabase Secret에만 저장, 클라이언트 노출 없음
- Webhook 수신 시 `X-Hub-Signature-256` HMAC 검증 필수
- RLS로 타 교회 티켓 격리
- 이슈 본문에 사용자 이메일/이름 미포함 (church + role만 기재)
- master만 GitHub 이슈 URL 노출 (일반 사용자는 내부 상태만)
