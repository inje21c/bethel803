# 27. 테넌트(교회) 격리 전수 테스트 절차

작성일: 2026-06-15
관련: 23(확장 계획 Phase 9-3 RLS 격리 테스트), 21(saas_church_scope)

## 1. 핵심 원칙

- 교회 경계(테넌트 격리)는 **데이터가 아니라 스키마**(RLS 정책 + SECURITY DEFINER 함수)가 결정한다.
- staging과 prod는 동일 마이그레이션(001~033)이므로 **staging 검증 결과 = prod 보증**.
- 따라서 격리 전수 테스트는 **staging에서 수행**한다. prod는 기존 벧엘 1교회로 읽기 검증만 보조.

## 2. 이메일 검증 우회

prod는 가입 시 이메일 확인이 켜져 있어 테스트 계정 대량 생성이 번거롭다.
→ `auth.admin.createUser({ email_confirm: true })` (service_role 키)를 쓰면
   **검증 메일 없이 즉시 확정 계정**을 만들 수 있다. staging/prod 공통.

`handle_new_user` 트리거 계약:
- 메타데이터 `church_name` → 새 교회 + 1구역 + master(active) 자동 생성
- 메타데이터 `district_id` → 해당 교회/구역 합류 (교회 첫 유저면 master, 아니면 member/pending)
- 임의 leader·추가 구역은 service_role UPDATE/INSERT로 마무리

## 3. 실행 순서 (staging)

```bash
# 1) 시드: 교회 A/B × (구역2 + 구역별 leader + member 2명)
SUPABASE_URL=<staging-url> \
SUPABASE_SERVICE_ROLE_KEY=<staging-service-role> \
node scripts/seed_tenant_fixtures.mjs
# → scripts/.tenant-fixtures.json 생성 (gitignore됨, 계정/비번 포함)

# 2) 격리 전수 검증 (anon 키로 각 계정 로그인 후 교차 접근 시도)
SUPABASE_URL=<staging-url> \
SUPABASE_ANON_KEY=<staging-anon> \
node scripts/test_tenant_isolation.mjs
# → PASS/FAIL 매트릭스 출력. FAIL 시 exit 1.

# 3) 정리
SUPABASE_URL=<staging-url> \
SUPABASE_SERVICE_ROLE_KEY=<staging-service-role> \
node scripts/seed_tenant_fixtures.mjs --teardown
```

prod에 실행하려면 시드 스크립트에 `--allow-prod` 필요(안전장치). 권장하지 않음.

## 4. 검증 대상 (전수)

- **church_id 직접 테이블**: districts, users, church_settings, churches, bible_reading_plans
- **district 경유**: prayer_requests, bible_studies, qt_contents, schedules, weekly_reports
- **user 경유**: qt_responses, study_answers, bible_reading_logs, bible_bookmarks, streaks,
  deep_meditations, notification_preferences, push_subscriptions
- **교회 스코프 RPC**: get_my_church_info, get_bible_reading_summaries,
  get_qt_district_summary, compute_weekly_report
- **역할 경계**: study_answers 타인 답변 비공개, role 자가 상승 차단(enforce_role_change_master_only),
  assign_my_district 타교회 배정 차단

판정: 교회 A 계정으로 교회 B 행이 1건이라도 보이거나, 타 교회 district_id로 RPC가 데이터를
반환하거나, member가 권한 상승에 성공하면 **FAIL(누수)**.

## 5. 주의

- 글로벌 공유 테이블(bible_books, bible_verses, daily_devotionals)은 격리 대상 아님(정상 공유).
- 픽스처 파일에는 계정/비번이 들어가므로 절대 커밋하지 않는다(.gitignore 처리됨).
- CI 편입 시: staging 대상으로 시드 → 검증 → 정리를 한 잡으로 묶고 service_role은 secret으로.
