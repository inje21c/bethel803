# 앱스토어 출시 전략 (iOS / Android)

작성일: 2026-06-22
대상: 벧엘구역 (React + Vite + Supabase PWA → 네이티브 앱 스토어 등록)

---

## 0. 요약 (TL;DR)

- 기술 경로: **Capacitor**로 기존 웹앱을 네이티브 셸로 래핑 (코드 99% 재사용, iOS·Android 동시).
- 가장 먼저 정할 것: **(1) 결제/후원을 앱에 넣을지** + **(2) 푸시 네이티브 전환(APNs/FCM)**.
- 리스크 1순위: Apple Guideline 4.2(단순 웹뷰 거절), 3.1.1(인앱결제 강제 / 후원 계좌 노출 제한).
- 출시 전 인프라 전환 필요: **Vercel Pro**, **커스텀 도메인**, (수익 발생 시) **사업자등록**.

---

## 1. 기술 경로 선택

| 방식 | 적합도 | 비고 |
|------|--------|------|
| **Capacitor (권장)** | 최적 | 기존 React 코드 재사용. 네이티브 푸시/카메라/딥링크 플러그인. iOS·Android 동시 |
| TWA (Android 전용) | 보조 | Play 스토어용 PWA 래퍼. 간단하나 iOS 불가, Apple 심사 회피 안 됨 |
| React Native 재작성 | 비권장 | 비용 대비 효과 없음 |

→ **Capacitor 단일 경로**로 iOS·Android 모두 커버.

## 2. 자주 묻는 핵심 질문 (Q&A)

### Q1. "후원하기"로 계좌번호를 앱에 노출해도 되나?
- **Apple**: 원칙적으로 막힘. 순수 기부라도 **Apple에 등록된 비영리단체(nonprofit)로 인증된 경우에만** 외부 결제(계좌이체) 허용. 미인증/영리 상태에서 계좌번호 텍스트 노출은 **Guideline 3.1.1 위반 → 거절 위험 높음**.
- **Google Play**: 기부에 관대. "기부임"을 명확히 표시하면 외부 계좌 안내 대체로 허용.
- **권장 대응**:
  - 후원 안내를 앱 내부가 아닌 **외부 브라우저(웹페이지)로 연결**.
  - 또는 교회를 **종교 비영리로 Apple에 인증**.
  - iOS 빌드에서만 후원 버튼을 숨기는 플랫폼 분기 처리도 흔한 패턴.

### Q2. Vercel 무료(Hobby) 계정으로 서비스해도 되나?
- Hobby 플랜 ToS는 **비상업적 용도 전용**. 다른 교회에 **유료 구독(SaaS) 판매 시 상업적 사용 → Pro($20/월) 필요**.
- 대역폭/서버리스 함수 실행 한도 존재 → 사용자 증가 시 병목.
- **정식 출시 전 Vercel Pro 전환 권장**.

### Q3. `*.vercel.app` 무료 도메인으로 괜찮나?
- 동작은 하지만 출시 단계에 부적합.
- **딥링크(Universal Links / App Links)는 도메인 루트에 소유권 검증 파일(apple-app-site-association, assetlinks.json)을 올려야 해서 커스텀 도메인이 사실상 필수.**
- OAuth 검수·브랜딩 신뢰도 측면에서도 필요. **연 1~2만원 커스텀 도메인 구매 권장.**

### Q4. 사업자 없는 개인도 앱스토어에 올릴 수 있나?
- **등록 자체는 개인 명의로 가능**: Apple Individual($99/년, 실명 노출), Google 개인 개발자($25, 1회).
- 무료 앱은 사업자등록번호 없이 출시 가능.
- **단, 구독/후원 등 수익 발생 시 한국 세법상 사업자등록이 사실상 필요**하고 Apple/Google 모두 세금·은행 정보 입력 요구.
- Google은 신규 개인 개발자에게 **20명 비공개 테스트 14일** 요건 부과(2023년 이후).

## 3. Phase A — 사전 준비 (계정 · 정책 · 결제 의사결정)

1. **개발자 계정 등록** (심사 전 미리, 승인에 며칠~수주):
   - Apple Developer Program: 연 $99. 조직 계정은 D-U-N-S 번호 필요.
   - Google Play Console: 1회 $25.
   - 가능하면 교회/단체 명의 권장.
2. **결제 정책 결정 (최대 리스크)**:
   - 앱 내 디지털 구독 판매 시 Apple은 **IAP(인앱결제, 30% 수수료) 강제**.
   - 회피 전략: **앱은 무료**로 풀고, 교회 유료 구독(B2B)은 **웹에서만 결제**. 앱은 "결제는 웹에서" 안내만. (B2B SaaS 표준 패턴, Apple 허용)
   - 후원은 Q1 참고 (외부 링크 또는 비영리 인증).
3. **Apple 4.2 거절 대비**:
   - 단순 웹뷰 래퍼는 "최소 기능 미달"로 거절 → **네이티브 푸시 알림**을 넣어 앱다운 가치 확보.

## 4. Phase B — Capacitor 도입 (개발)

4. Capacitor 설치 및 iOS/Android 프로젝트 생성 (`@capacitor/core`, `@capacitor/ios`, `@capacitor/android`).
5. **푸시 알림 네이티브화**: 현재 Web Push(VAPID) → iOS는 **APNs**, Android는 **FCM** 필요.
   - `@capacitor/push-notifications` + Firebase 프로젝트 연동.
   - 네이티브 토큰을 기존 `push_subscriptions` 테이블에 함께 저장하도록 확장 (push.ts 수정).
6. **OAuth 점검**: 카카오/구글 로그인이 인앱 WebView에서 차단될 수 있음.
   - `@capacitor/browser`(시스템 브라우저) 또는 네이티브 SDK 사용.
   - 리디렉트 URL에 커스텀 스킴(예: `com.bethel.app://`) 추가.
7. **딥링크**: 초대 링크(`/join?d=<district_id>`) → Universal Links / App Links 설정 (커스텀 도메인 필요).
8. 앱 아이콘·스플래시(`@capacitor/assets`), 세이프에어리어(노치) CSS, 상태바 처리.

---

## 5. Phase C — 스토어 자산 준비

9. **필수 자산**:
   - 앱 아이콘(1024px), 스크린샷(iPhone 6.7"/6.5", iPad, Android 폰/태블릿 각 사이즈).
   - 앱 이름/부제/설명, 키워드, 카테고리(교육 또는 라이프스타일).
   - **개인정보처리방침 URL** (`/privacy` 기존 페이지 사용 가능).
   - ATT 문구, 데이터 수집 신고(Apple Privacy Nutrition Label / Google Data Safety).
10. **심사용 데모 계정**: 심사관 로그인용 데모 교회 계정 준비 (베타 테스터 인프라 재활용).

---

## 6. Phase D — 제출 · 심사 · 출시

11. **Android 먼저 제출** (심사 빠르고 관대) → 피드백 반영 후 iOS 제출.
12. 단계적 출시: Play는 **내부→비공개→프로덕션**, Apple은 **TestFlight** 베타 먼저.
13. 출시 후: 크래시 모니터링, 네이티브 셸 업데이트는 스토어 심사 필요(웹 자산 OTA 갱신 가능하나 Apple 정책 주의).

## 8. 권장 진행 순서 (의사결정 우선)

가장 먼저 막히는 두 지점:
1. **결제/후원을 앱에 넣을지** (IAP 강제 + 후원 계좌 제한)
2. **푸시 네이티브 전환** (APNs/FCM)

이 둘을 확정하면 나머지는 기계적으로 진행 가능.
