# 26. QT·구역성경공부 범용화 설계 (SaaS 전환 핵심 과제)

작성일: 2026-06-12
관련: 23(확장 계획 Phase 10), 25(범용화 3단 흡수 원칙)

## 1. 문제 정의

QT와 구역성경공부는 벧엘교회에 커스터마이징된 두 기능이며 SaaS 전환의 핵심 장애물이다.

| 기능 | 벧엘 전용 요소 | 범용화 난이도 |
|------|---------------|--------------|
| QT | 매일성경(성서유니온) 스크래핑 + AI 요약 파이프라인 | 높음 — 콘텐츠 소스 자체가 외부 저작물 |
| 구역성경공부 | 주보 PDF 파싱(bethel.or.kr) | **낮음** — 수동 등록(createBibleStudy)은 이미 존재 |

### 저작권 제약 (중요)

- 매일성경 스크래핑 콘텐츠는 성서유니온 저작물. 벧엘 단독 사용과 달리
  **타교회 재배포(SaaS)는 명확한 저작권 침해 리스크** → scraped 모드는 벧엘 전용으로 격리
- AI가 성경 해설을 생성하는 것은 신학적 책임 문제 → **AI는 질문 생성까지만**
- 자체 성경 DB(022 구조화 리더용 시드)는 외부 의존 없음 → simple 모드의 구절 소스

## 2. 설계: qt_mode 3종 (택일이 아닌 교회별 설정)

`church_settings.qt_mode`:

| 모드 | 콘텐츠 | AI 역할 | 사용자 흐름 |
|------|--------|---------|------------|
| `scraped` | 매일성경 스크래핑 (현행) | 요약+질문 (현행) | 현행 그대로 |
| `admin` | 마스터/구역장이 날짜별 등록 (본문 참조+묵상 포인트+질문) | 질문 보조 생성 (선택) | 현행과 동일 UI |
| `simple` | 자체 성경 DB에서 그날 구절 1개 (읽기 플랜 순서 또는 큐레이션 목록) | 질문만 선택 생성, 해설 없음 | 구절 노출 → 묵상 체크 (qt_responses.is_completed) |

- 신규 교회 디폴트: `simple` (설정 없이 즉시 동작 — 25번 문서의 컨벤션 원칙)
- 깊은 묵상(4단계): simple/admin 모드에서는 "본문 요약" 자리에 본문 그대로 노출,
  질문은 AI 생성(캐시) 또는 기본 4문항 — 현 폴백 구조 재사용

## 3. 구역성경공부 범용화

1. 수동 등록 UI는 존재 → "공부 만들기" 흐름으로 다듬기 (제목/본문/질문 템플릿)
2. 주보 PDF 파싱 = 기능 모듈 플래그 `bulletin_parsing` 기본 OFF (URL 패턴은 교회 설정)
3. 소모임(구역) 단위 배정은 district 구조가 이미 지원

## 4. 개발 분기 전략: "특정 아이디 분기" 대신

특정 아이디 하드코딩 분기는 handle_new_user의 이메일 하드코딩과 같은 부채를 재생산한다.
대신 이미 갖춰진 인프라를 쓴다:

```
feat/qt-saas 브랜치 push
  → Vercel Preview (자동) — staging Supabase 연결됨 (021 churches 적용 상태)
  → staging 제2교회 (docs/staging_second_church_test.sql) 계정으로 접속
  → church_settings.qt_mode='simple' 교회에서 신규 교회 경험 검증
  → 벧엘 prod는 코드·데이터 모두 무영향
```

- 코드 분기점은 **교회 설정값(qt_mode, 모듈 플래그)** — 테스트용 분기가 아니라
  그대로 SaaS 출시 구조이므로 나중에 걷어낼 코드가 없다
- prod의 벧엘은 021 적용 전까지 church_settings가 없으므로,
  프론트는 "설정 없음 → scraped(현행)" 폴백으로 동작 (기존 사용자 무변화)

## 5. 단계별 작업 (staging 우선)

| 순서 | 작업 | 비고 |
|------|------|------|
| 1 | 027: church_settings 테이블 (church_id FK, qt_mode, module flags, bulletin_url_pattern 등) | staging 먼저 |
| 2 | qt_contents에 church_id 추가 (기본 벧엘 교회 귀속) + RLS 교회 스코프 | scraped 데이터는 벧엘 소유로 격리 |
| 3 | simple 모드: 일별 구절 선정(성경 DB) + 묵상 체크 흐름 | 신규 교회 디폴트 |
| 4 | admin 모드: 관리자 QT 등록 UI (AdminDashboard 탭) | AI 질문 보조 버튼 포함 |
| 5 | 성경공부: bulletin_parsing 플래그 + 공부 만들기 UI 정리 | |
| 6 | staging 제2교회로 simple/admin 모드 E2E 검증 | preview URL |
| 7 | (prod 021 적용 이후) 벧엘 scraped 모드 명시 설정 | Phase 9 완료 후 |

## 6. 결정 사항 요약

- 세 가지 QT 안(관리자 등록/AI 질문만/심플 체크)은 **모두 채택** — qt_mode로 공존
- AI 해설 생성 금지, 질문 생성만 (모드 무관)
- scraped 모드는 저작권상 벧엘 전용 격리
- 개발·검증은 특정 아이디 분기가 아니라 **feat 브랜치 → preview → staging 제2교회** 경로
