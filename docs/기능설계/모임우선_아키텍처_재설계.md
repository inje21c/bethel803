# Group-First Architecture / 모임 우선 아키텍처

> Public document / 공개 문서
> First designed / 최초 설계: 2026-06-21
> Last reviewed / 마지막 점검: 2026-07-07

## English Summary

Group-First Architecture is the product and data model direction used by bethel803. Instead of forcing every new user to understand a church-level administration model first, the service starts from the small group: create a group, invite members, and let the leader care for that group.

Church-level administration still exists, but it is treated as a higher-level setup path for organizations, not the default everyday path for members.

## 한국어 요약

모임 우선 아키텍처는 bethel803의 제품 방향과 데이터 모델 방향입니다. 새 사용자가 처음부터 교회 단위 관리 구조를 이해하도록 요구하지 않고, 먼저 모임을 만들고 구성원을 초대하며 리더가 그 모임을 돌보는 흐름을 기본값으로 삼습니다.

교회 단위 관리는 여전히 존재하지만, 일반 구성원의 일상 흐름이 아니라 조직 단위 도입과 상위 관리 흐름으로 분리합니다.

## 1. Problem / 문제의식

English:

Early versions of the product exposed a three-level model too directly: church, district/group, and user. That structure was technically reasonable, but it did not match how a casual user thinks when joining or starting a small group. The result was unnecessary onboarding complexity and a higher risk of choosing or joining the wrong scope.

한국어:

초기 구조는 교회, 구역/모임, 사용자라는 3단계 모델을 사용자에게 너무 직접적으로 드러냈습니다. 기술적으로는 자연스러운 구조였지만, 일반 사용자가 소그룹을 시작하거나 참여할 때 떠올리는 사고방식과는 맞지 않았습니다. 그 결과 온보딩이 복잡해지고 잘못된 범위를 선택하거나 참여할 위험이 생겼습니다.

## 2. Design Goal / 설계 목표

| Goal | Korean |
| --- | --- |
| Start from the group, not the institution | 교회 조직보다 모임 경험을 먼저 둔다 |
| Make the first action obvious | 첫 행동을 명확하게 만든다 |
| Keep leader permissions narrow | 리더 권한을 자기 모임 범위로 제한한다 |
| Preserve church-level administration for real organizations | 실제 조직 도입을 위한 교회 단위 관리는 유지한다 |
| Use RLS as the final data boundary | 최종 데이터 경계는 RLS로 강제한다 |

## 3. Adopted Model / 채택한 모델

### 3.1 Community Container / 커뮤니티 컨테이너

English:

Free or self-started groups can live under a shared community container. Users experience this as "my group," while the system keeps using the existing church and district data model internally.

한국어:

무료 또는 셀프 개설 모임은 공통 커뮤니티 컨테이너 아래에 둘 수 있습니다. 사용자는 이를 "내 모임"으로 경험하지만, 시스템 내부에서는 기존 교회와 구역 데이터 모델을 유지합니다.

### 3.2 Leader Scope / 리더 범위

English:

A group creator becomes the leader of that group, not the master of an entire church-level organization. The leader can manage members and content inside the group scope.

한국어:

모임 개설자는 전체 교회 단위의 마스터가 아니라 해당 모임의 리더가 됩니다. 리더는 자기 모임 안에서 구성원과 콘텐츠를 관리합니다.

### 3.3 Church-Level Administration / 교회 단위 관리

English:

Church-level setup is reserved for organization-level use cases. It can be handled through superadmin support, migration, or a dedicated business setup flow.

한국어:

교회 단위 셋업은 조직 단위 사용 사례를 위한 경로로 남깁니다. 슈퍼어드민 지원, 데이터 이전, 별도 도입 흐름을 통해 처리할 수 있습니다.

## 4. Why This Model / 이 모델을 선택한 이유

| Consideration | Group-first direction |
| --- | --- |
| User mental model / 사용자 사고방식 | People usually think "my group" before "my church tenant." 일반 사용자는 보통 "내 교회 테넌트"보다 "내 모임"을 먼저 생각합니다. |
| Permission safety / 권한 안정성 | Leaders stay scoped to their own group. 리더 권한이 자기 모임에 머뭅니다. |
| Reuse of existing RLS / 기존 RLS 재사용 | Existing church/district RLS boundaries can remain useful. 기존 교회/구역 RLS 경계를 계속 활용할 수 있습니다. |
| Product positioning / 제품 포지셔닝 | Free groups and organization-level deployments can be separated. 무료 모임과 조직 단위 도입을 분리할 수 있습니다. |
| Operational clarity / 운영 명확성 | Superadmin operations handle exceptional transitions. 예외적 전환은 슈퍼어드민 운영으로 처리합니다. |

## 5. Signup and Join Flow / 가입과 참여 흐름

### Create a Group / 모임 만들기

```text
User starts signup
 -> create group
 -> new district/group is created under the community container
 -> creator becomes leader
 -> invite link can bring members into the group
```

```text
사용자가 가입 시작
 -> 모임 만들기
 -> 커뮤니티 컨테이너 아래 새 모임 생성
 -> 개설자는 리더가 됨
 -> 초대 링크로 구성원 참여
```

### Join by Invitation / 초대 링크 참여

```text
Invitation link
 -> target group lookup
 -> signup or login
 -> member joins the invited group
 -> leader approval if configured
```

```text
초대 링크
 -> 대상 모임 확인
 -> 가입 또는 로그인
 -> 초대받은 모임에 구성원으로 참여
 -> 설정에 따라 리더 승인
```

## 6. Data Isolation / 데이터 격리

The architecture relies on these boundaries:

- `church_id` separates organization-level scope.
  `church_id`는 조직 단위 범위를 나눕니다.
- `district_id` separates group-level scope.
  `district_id`는 모임 단위 범위를 나눕니다.
- RLS policies enforce final access control.
  RLS 정책이 최종 접근 제어를 강제합니다.
- UI checks improve experience but are not treated as the security boundary.
  UI 검사는 사용자 경험을 돕지만 보안 경계로 보지 않습니다.

## 7. Graduation Path / 모임 승격 경로

English:

If a free community group needs to become part of a formal church deployment, the group can be moved or graduated through a controlled superadmin operation. This keeps everyday signup simple while preserving a path for organization-level adoption.

한국어:

무료 커뮤니티 모임이 정식 교회 단위 도입으로 전환되어야 할 경우, 슈퍼어드민의 통제된 작업을 통해 모임을 이동하거나 승격할 수 있습니다. 이렇게 하면 일반 가입 흐름은 단순하게 유지하면서도 조직 단위 도입 경로를 보존할 수 있습니다.

Representative operations / 대표 운영:

- Move a group into an existing church scope.
  모임을 기존 교회 범위로 이동합니다.
- Graduate a group into a new church-level organization.
  모임을 새 교회 단위 조직으로 승격합니다.
- Promote or remap leadership only through controlled admin flows.
  리더십 승격과 매핑은 통제된 관리 흐름으로만 처리합니다.

## 9. Public Lessons / 공개 가능한 설계 교훈

English:

- Data models that are technically correct can still expose the wrong product model.
- Self-service onboarding should begin with the object users understand.
- Multi-tenant systems should avoid public enumeration of other tenant scopes.
- Role names should reflect what users can safely do, not only the database hierarchy.

한국어:

- 기술적으로 맞는 데이터 모델도 제품 경험으로는 맞지 않을 수 있습니다.
- 셀프서비스 온보딩은 사용자가 이해하는 대상에서 시작해야 합니다.
- 멀티테넌트 시스템은 다른 테넌트 범위가 공개적으로 열거되지 않도록 해야 합니다.
- 권한 이름은 DB 계층만이 아니라 사용자가 안전하게 할 수 있는 일을 반영해야 합니다.
