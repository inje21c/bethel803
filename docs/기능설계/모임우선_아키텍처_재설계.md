# 모임 우선(Group-First) 아키텍처 재설계

작성일: 2026-06-21
배경: 베타 오픈 첫날, 사용자가 교회 등록 대신 구역원으로 가입 → 전역 첫 활성 구역(테스트구역)에 자동 배정되는 사고 발생. 3단계(교회→구역→구역원) 구조가 일반 사용자 멘탈모델과 맞지 않음.

---

## 1. 현재 구조 진단

### 1.1 3단계 모델
- churches → districts → users (3-tier 멀티테넌트)
- 셀프가입 시 `church_name` 입력 → 교회+구역+master 자동 생성 (handle_new_user)
- 일반 가입자는 구역(district)을 골라 member로 합류

### 1.2 어제 사고의 두 누수 지점 (코드 레벨)
1. **Login.tsx 회원가입 탭 "소속 구역 선택" 드롭다운** (`regDistrictId`)
   - `districts_select_active` RLS가 모든 활성 교회의 모든 활성 구역을 anon/authenticated에 노출
   - 낯선 사람이 임의 교회의 구역을 선택해 가입 가능 = 테넌트 경계 침범
2. **handle_new_user fallback** (032)
   - church_name·district_id 둘 다 없으면 `ORDER BY d.created_at LIMIT 1` → 전역 첫 활성 구역 자동 배정

### 1.3 이미 잘 되어 있는 것 (재설계의 토대)
- RLS 데이터 정책이 일관되게 `is_church_master(church_id) OR (is_church_leader(church_id) AND district_id = get_my_district_id())`
- **리더는 같은 교회 안에서도 자기 구역만 접근** → 검증됨(테넌트 격리 25 PASS)
- 라우팅 분리: `/admin`=LeaderRoute, `/districts`=MasterRoute (구역관리는 이미 master 전용)

## 2. 목표 구조 (Option A — 단일 커뮤니티 컨테이너)

### 2.1 핵심 아이디어
- **"일반사용자교회"(커뮤니티 컨테이너)** 라는 고정 교회 1개를 둔다.
- 일반 사용자가 랜딩에서 "모임 만들기" → 이 컨테이너 **밑에 새 district(모임) 생성**, 개설자는 그 모임의 **leader**.
- leader는 RLS상 자기 모임만 봄 → 사용자 입장에선 "내 모임 하나"만 존재하는 2단계처럼 보임.
- **"구역관리(여러 구역)" 개념은 master(=슈퍼어드민/B2B 교회)에게만 노출.**
- 교회(church)는 슈퍼어드민이 B2B 영업으로 직접 생성하는 희소·유료 단위로 포지셔닝.
- 모임이 "교회로 통합/승격" 원하면 슈퍼어드민이 수동으로 연결(graduate).

### 2.2 왜 Option A인가 (vs Option B = 모임마다 교회 1개)
| 항목 | A: 단일 컨테이너 | B: 모임=교회 |
|---|---|---|
| 격리 모델 | 기존 leader-구역스코프 그대로 (신규 코드 0) | church-master 단일구역, UI 게이팅 다수 필요 |
| 사용자 멘탈모델 | 모임만 보임(2단계) ✓ | master라 구역관리/교회설정 노출 위험 |
| 교회 = 희소 B2B | 부합 ✓ | 가입마다 교회 양산(포지셔닝 어긋남) |
| 구현 리스크 | 낮음(검증된 경로) | 높음(단일구역 교회 master 누수 재감사) |
| **단점** | church_settings(modules/plan/ui_mode/qt_mode)가 컨테이너 모든 모임에 공유 | 모임별 설정 독립 가능 |

> 주의: **베타 플래그(beta_flags)는 이 단점과 무관.** beta_flags는 church 단위가 아니라 **서비스 전역**(050)이라 컨테이너든 벧엘이든 모두에게 동일 적용된다. 공유 한계는 hasModule 판정의 마지막 단계인 교회별 `church_settings.modules` 라이선스(베타 OFF 후 적용)에만 해당한다. 벧엘은 plan='legacy'라 항상 전부 열려 영향 없음.

→ **A 채택.** 단점(설정 공유)은 무료 커뮤니티 티어에 오히려 적합(균일 무료 기능셋). 개별 과금/커스터마이즈가 필요해지면 graduate로 자기 교회 승격.

## 3. 데이터 모델 변경

### 3.1 커뮤니티 컨테이너 교회 생성 (신규 마이그레이션)
- 고정 UUID: `00000000-0000-4100-a000-000000000002` (벧엘=...001)
- name: "커뮤니티" (또는 "모임 모음"), slug: `community`
- status: `active`, plan: `community`(plan CHECK에 추가) 또는 `free`+trial_ends_at NULL
- **트라이얼 만료 없음** (billing_status='manual', trial_ends_at=NULL) → 만료 배너/게이팅 미적용
- **master 행 없음** → 일반 RLS로는 누구도 컨테이너 전체를 못 봄. 운영은 슈퍼어드민 SECURITY DEFINER RPC로만.

### 3.2 plan enum 확장
`churches_plan_check`에 `community` 추가: `('legacy','free','community','starter','standard','premium')`

## 4. handle_new_user 재설계

분기 우선순위:
1. **`district_id` 있음 (초대 링크)** → 해당 구역 member/pending (기존 유지). 컨테이너는 비어있지 않으므로 "첫 사용자=master" 분기 미발동.
2. **`create_group=true` (+ group_name)** → 커뮤니티 컨테이너 밑에 새 district 생성, role=`leader`, status=`active`. (master 절대 아님)
3. **bare 가입 (메타데이터 없음)** → **전역 첫 구역 fallback 제거.** 대신 #2와 동일 처리: `group_name = COALESCE(name)+'의 모임'`으로 자기 모임 자동 생성. → orphan/전역배정 사고 원천 차단.
4. **`church_name` 있음** → 공개 셀프가입에서는 제거. 교회 생성은 슈퍼어드민 전용 경로로 이동.

핵심: **비초대 가입의 기본 동작 = "내 모임을 만들고 내가 리더"**. 전역 첫 구역 배정은 영구 폐기.

## 6. Graduate 경로 (모임 → 교회 승격/통합)

`prevent_district_church_change_with_users` 트리거가 "사용자 있는 구역의 church 이동"을 막음 → 슈퍼어드민 전용 SECURITY DEFINER RPC 필요.

```
move_district_to_church(p_district_id, p_target_church_id, p_new_master_user_id)
  - district.church_id 이동 + 해당 구역 모든 users.church_id 동반 이동 (트리거 우회: 함수 내 일괄 처리)
  - 필요 시 개설자 leader → master 승격(044 교훈: 신규 master 먼저 승격, 구 master 강등 나중)
  - 슈퍼어드민만 실행
```
케이스:
- **단독 승격**: 모임 1개를 새 교회로 분리 → 개설자가 그 교회 master.
- **기존 교회로 통합**: 여러 모임을 한 교회로 합침 → 슈퍼어드민이 수동 매핑.

## 9. 벧엘교회 무영향 보장

- 벧엘(`...001`)은 그대로 독립 교회. master=cmhyun 유지. 데이터 RLS 변경 없음.

## 10. 데이터 정리 (사고 수습)

- 어제 테스트구역(`ba2ea0f1…`, 벧엘 밑)에 잘못 들어온 구역: **052에서 커뮤니티 컨테이너로 통째 이동** 결정.
- 구성원K(`UUID-K`, 어제 타구역 pending 가입, 테스트 참여자): **052에서 테스트구역 구역원으로 이동 후 커뮤니티로 동반 이동**(role=member/status=pending 유지). 타구역 정상 가입이 아니라 테스트 참여자로 판단되어 정리.
- **베타 중 셀프가입으로 생성된 트라이얼 교회 3개 → 전부 삭제 확정 (2026-06-22).**
  - 대상: 테스트교회(`UUID-T`, master 테스트교회관리자, 성경공부 1), 베타교회A(`UUID-A`, master 테스터A, 일정 2), 베타교회B(`UUID-B`, master 테스터H, 내용 없음). 셋 다 master 1명·다른 구성원 없는 빈 테스트 껍데기.
  - 사유: 구식 셀프가입 잔재이며 새 모델(교회=슈퍼어드민 B2B 전용)과 불일치 + DistrictPicker 전역 누수원.
  - **삭제 절차** (3단계 순서 중요):
    1. 소프트딜리트: `churches.deleted_at` 설정 (3개).
    2. `hard_delete_church_superadmin(church_id)` RPC 호출 → CASCADE로 districts/users/콘텐츠 삭제.
    3. master 3명의 **auth.users 로그인 계정은 RPC가 안 지움** → 별도 Edge Function/Admin API(service_role)로 삭제.
  - ⚠️ 주의: 베타교회B master "테스터H"은 벧엘 킨텍스장성남의 "테스터H"(member)과 **별개 auth 계정**. 베타교회B 삭제는 벧엘 계정 미접촉. 단 auth 계정 삭제 전 이메일 확인 권장.
  - 실행 시점: 구현 착수 전/중 일회성 정리(스크립트 또는 마이그레이션). DistrictPicker 누수 차단(12.7-1)과 함께 처리하면 재유입 방지.

## 12. 영향 분석 — 5대 흐름 + 공통 (2026-06-22)

전수 추적 결과. severity: 🔴 반드시 수정(누수/깨짐) · 🟡 조정 필요(UX/카피/로직) · 🟢 안전/이미 처리됨

### 12.0 공통 — 가장 큰 누수 (두 곳)
- 🔴 **`get_active_districts_with_church()` RPC (049)** — anon+authenticated에게 **전 테넌트의 모든 교회·구역 이름** 나열. 커뮤니티 컨테이너 생성 시 모든 모임 이름이 익명에게 노출. 사용처는 `getActiveDistricts → DistrictPicker` 단 한 갈래 → **DistrictPicker 제거 시 RPC도 폐기 가능**.
- 🔴 **`districts_select_active` RLS (021)** — 테이블 레벨에서도 anon이 모든 활성 구역 SELECT 가능. Join.tsx는 id 단건 조회만 필요 → 열거 차단 방향 재검토.

### 12.1 랜딩 (Landing.tsx)
- 🟡 모든 CTA가 `/signup/church`. → "모임 만들기"(신규 `/signup`)와 B2B 소개 페이지로 분기. 순수 프론트.

### 12.2 회원가입
- 🔴 **Login 회원가입 탭 + DistrictPicker** — 전역 목록 드롭다운(어제 사고 직접 원인). **PendingApproval.tsx에서도 동일 사용** → 두 곳 모두 제거/전환.
- 🔴 **handle_new_user 전역-첫-구역 fallback (032)** — 폐기 확정(4장).
- 🟡 **ChurchSignup.tsx** — 공개 "교회 등록" = 트라이얼 교회 양산(이미 3개: 테스트/단비/철민). 슈퍼어드민 내부용 격하 + `create_group` 경로 신규.
- 🟢 **Join.tsx** — 초대 링크는 구역 id 단건 동작(열거 없음). 유효. 카피만 조정.

### 12.4 이용안내 (UserManual.tsx)
- 🟡 전반이 교회/3단계/마스터 프레이밍 → 커뮤니티 사용자에 부정확:
  - "체험판 안내"(L233): 커뮤니티는 trialing 아님 → 무관.
  - "마스터 권한 이관 — 교회 전체 관리 권한"(L265): 커뮤니티엔 master 개념 없음.
  - 정식 도입 건의 카피(L460-461): 커뮤니티는 "우리 교회의 플랜에서는 제한"으로 떨어져 "교회 플랜" 언급 → 약간 어긋남.
  - 🟢 `isLeaderOrMaster` 게이팅(L319)은 리더에 구역장 가이드 노출 → 커뮤니티 리더에 적합.
  - → plan/컨테이너 기준 카피 분기 또는 톤 완화 1패스.

### 12.5 회원탈퇴 (delete-account/index.ts)
- 🔴 **리더 탈퇴 = 본인만 즉시 삭제(else 분기)**. 커뮤니티 모임 개설자=리더 → 탈퇴 시 **리더 부재 모임**(승인·관리 불가). master 로직처럼 "활성 구성원 있으면 차단" 또는 자동 승계 필요.
