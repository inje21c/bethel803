# 성경 북마크, 1년 1독, DB 운영 전략

## 1. 현재 결정

2026-06-07 기준으로 성경 기능은 다음 순서로 진행한다.

1. 북마크 저장/보기 개선을 먼저 production에 반영한다.
2. 그 다음 성경읽기 챌린지를 개발한다.
3. 성경읽기 챌린지 개발 이후, 개발/운영 DB 구조 전환을 별도 작업으로 진행한다.

이 순서를 택한 이유는 현재 preview 브랜치와 production 앱이 서로 다른 Supabase 프로젝트를 바라보고 있어, 성경 본문 같은 기준 데이터가 preview DB에 없으면 preview 테스트가 막히기 때문이다. 우선 사용자에게 필요한 북마크 기능은 production DB 기준으로 검증 가능하게 반영하고, DB 운영 전략은 기능 안정화 후 정리한다.

## 2. 북마크 기능 기준

북마크는 개인별 절 저장 기능이다.

### 2.1 사용자 흐름

- 성경 본문에서 절 옆 북마크 아이콘을 누르면 저장한다.
- 이미 저장된 절을 다시 누르면 북마크를 해제한다.
- `북마크` 탭에서 저장한 절 목록을 확인한다.
- 저장 목록에는 `책 장:절`과 본문 미리보기를 표시한다.
- `보기`를 누르면 해당 본문 위치로 이동한다.
- `삭제`를 누르면 개인 북마크에서 제거한다.

### 2.2 데이터 기준

사용 테이블:

- `bible_books`
- `bible_verses`
- `bible_bookmarks`

`bible_books`, `bible_verses`는 성경 기준 데이터이므로 환경별 사용자 데이터와 분리해서 본다. `bible_bookmarks`는 개인 데이터이므로 RLS로 본인 데이터만 CRUD 가능해야 한다.

## 3. 1년 1독 챌린지 설계 방향

1년 1독은 단순히 창세기부터 계시록까지 순서대로 읽는 방식으로 만들지 않는다. 이 방식은 중간 구간에서 이탈률이 높을 수 있으므로, 기본 플랜은 구약과 신약을 병행하는 방식으로 한다.

### 3.1 기본 플랜

하루 분량은 다음 형태를 기본으로 한다.

```text
구약 2-3장
신약 1장
```

예:

```text
Day 1
창세기 1-2장
마태복음 1장

Day 2
창세기 3-4장
마태복음 2장
```

성경 전체 장수:

- 구약: 929장
- 신약: 260장
- 전체: 1,189장

365일 기준 하루 평균은 약 3.26장이다. 따라서 대부분의 날은 구약 2-3장과 신약 1장을 배치하고, 신약이 끝난 뒤에는 구약 잔여 분량을 조정한다.

### 3.2 UI 방향

`성경` 메뉴는 최종적으로 다음 탭 구조를 목표로 한다.

```text
본문 읽기
북마크
1년 1독
읽기 기록
```

`1년 1독` 탭 예시:

```text
오늘의 1년 1독
Day 42

구약
출애굽기 16-18장

신약
마태복음 22장

총 4장

[본문으로 읽기]
[읽었어요]
[직접 기록]
```

### 3.3 기록 원칙

- `읽었어요`는 챌린지 분량을 빠르게 기록하는 보조 수단이다.
- 사용자가 앱 밖에서 읽거나 몰아서 읽을 수 있으므로 수동 입력은 계속 유지한다.
- `읽었어요`를 누르면 해당 day의 총 장수를 기존 `bible_reading_logs`에 자동 반영한다.
- 동시에 챌린지 진행 테이블에 완료 상태를 저장한다.
- 같은 day는 중복 완료되지 않게 한다.

## 4. 1년 1독 DB 초안

### 4.1 플랜

```text
bible_reading_plans
- id
- name
- description
- duration_days
- is_active
- created_at
```

### 4.2 날짜별 구간

하루에 구약/신약 여러 구간이 들어갈 수 있으므로 item은 day 단위 안에서 sequence를 가진다.

```text
bible_reading_plan_items
- id
- plan_id
- day_number
- sequence
- testament
- book_id
- chapter_from
- chapter_to
- chapter_count
```

예:

```text
Day 1, sequence 1, old, 창세기 1-2
Day 1, sequence 2, new, 마태복음 1
```

### 4.3 개인 진행

```text
bible_reading_challenge_progress
- id
- user_id
- plan_id
- day_number
- completed_at
- chapters_logged
- reading_log_id
- created_at
```

권장 제약:

```text
UNIQUE(user_id, plan_id, day_number)
```

## 5. 개발/운영 DB 전략

현재는 Supabase 무료 프로젝트 수 제한 때문에 production과 preview가 서로 다른 Supabase 프로젝트를 바라보는 구조가 생겼다. 이 구조는 작은 기능에는 도움이 되지만, 성경 본문처럼 큰 기준 데이터가 들어간 기능에서는 preview DB에 동일 seed가 없으면 테스트가 막힌다.

### 5.1 단기 방침

- 북마크 개선은 production에 먼저 반영한다.
- 1년 1독 챌린지는 현재 개발 브랜치에서 구현한다.
- 필요 시 production DB의 성경 기준 데이터로 테스트한다.

### 5.2 중기 전환 방향

무료 Supabase 제약 안에서는 DB 프로젝트를 환경별로 계속 늘리기 어렵다. 따라서 다음 방향을 검토한다.

```text
Production app -> 운영 Supabase
Preview app    -> 같은 Supabase 또는 공용 기준 데이터가 있는 Supabase
Local dev      -> 필요 시 local Supabase 또는 제한된 운영 테스트 계정
```

핵심은 DB 프로젝트 분리보다 데이터 영역 분리다.

공용 기준 데이터:

- `bible_books`
- `bible_verses`
- `bible_reading_plans`
- `bible_reading_plan_items`

개인/운영 데이터:

- `bible_bookmarks`
- `bible_reading_logs`
- `bible_reading_challenge_progress`
- `prayer_requests`
- `schedules`
- `bible_studies`

### 5.3 preview 격리 방식

SaaS 전환 구조의 `church_id`, `district_id`를 활용해 preview 전용 church/district를 둔다.

예:

```text
production church: 실제 벧엘 운영
preview church: preview 테스트
```

preview 앱은 preview church/test user만 사용하게 한다. 이렇게 하면 같은 Supabase 프로젝트를 쓰더라도 운영 데이터 오염을 줄일 수 있다.

### 5.4 주의할 작업

다음 작업은 운영 DB에서 바로 하지 않고 별도 리허설 후 반영한다.

- RLS 정책 대수정
- Auth 구조 변경
- 대량 삭제/정산/일괄 업데이트
- church scope 대규모 마이그레이션
- 1년 1독 진행 데이터 구조 확정 전 변경

## 6. 향후 작업 순서

1. 북마크 production 반영 완료 확인
2. 1년 1독 DB 마이그레이션 작성
3. 구약/신약 병행형 365일 seed 생성
4. `성경` 메뉴에 `1년 1독` 탭 추가
5. `본문으로 읽기`, `읽었어요`, `직접 기록` 구현
6. 챌린지 완료 시 `bible_reading_logs` 자동 기록 연동
7. preview/production DB 운영 전략 결정
8. church/district 기반 preview 격리 정리

## 7. 롤백 기준

북마크 기능은 기존 성경 본문/읽기 기록과 분리되어 있다. 문제가 생기면 `북마크` 탭과 북마크 API 변경만 되돌려도 기존 성경 본문 읽기와 수동 읽기 기록은 유지된다.

1년 1독은 별도 테이블과 별도 탭으로 추가한다. 문제가 생기면 챌린지 탭을 숨기고 수동 기록을 계속 사용하게 한다.
