# production 리허설 순서

## 1. 문서 목적

이 문서는 실제 운영 절체 전에 `Vercel Production + Supabase prod` 조합을 검증하는 실행 순서를 정리한다.

이 문서의 목적은 다음과 같다.

- preview/staging 성공을 production 리허설로 연결한다.
- 절체 직전의 확인 순서를 고정한다.
- 한 번에 너무 많은 것을 바꾸지 않도록 범위를 제한한다.

## 2. 리허설 범위

production 리허설에서 확인할 것은 `현재 운영 기능의 동등성`이다.

이번 리허설에 포함:

- 로그인
- 권한
- 성경공부
- 기도제목
- 일정
- 첨부 업로드
- 관리자 기능

이번 리허설에 제외:

- 웹푸시 실발송
- AI 추가 기능
- `013` 기반 신규 알림 구조

## 4. 실행 순서

### 4.1 production 배포

1. production 대상으로 배포 실행
2. 배포 성공 여부 확인
3. production URL 접속 확인

체크:

- [ ] 첫 화면 정상
- [ ] 정적 asset 404 없음
- [ ] 콘솔 치명적 오류 없음

### 4.2 로그인

1. `master` 계정 로그인
2. 로그아웃
3. `leader` 계정 로그인
4. 로그아웃
5. `member` 계정 로그인

체크:

- [ ] 로그인 성공
- [ ] 새로고침 후 세션 유지
- [ ] `/login`으로 튀지 않음
- [ ] 로그아웃 후 자동 재로그인 안 됨

### 4.3 역할별 접근

`master`

- [ ] `/dashboard`
- [ ] `/admin`
- [ ] `/districts`

`leader`

- [ ] `/dashboard`
- [ ] `/admin`
- [ ] `/districts` 차단

`member`

- [ ] `/dashboard`
- [ ] `/admin` 차단

### 4.4 성경공부

1. 성경공부 목록 확인
2. 기존 발행본 상세 확인
3. 관리자에서 원본 목록 확인
4. `내 구역 수정본 만들기` 확인
5. 수정본 저장
6. 발행 전환

체크:

- [ ] 발행본 조회 정상
- [ ] 구역 분리 정상
- [ ] 원본/수정본 구조 정상

### 4.5 기도제목 / 일정 / 첨부

- [ ] 기도제목 조회
- [ ] 일정 조회
- [ ] 일정 첨부 업로드
- [ ] 첨부 링크 열림

### 4.6 관리자 기능

- [ ] 승인 대기 목록 확인
- [ ] 구역원 승인
- [ ] 역할 변경
- [ ] 주간 보고 조회

## 5. 리허설 실패 시 처리

어느 한 항목이라도 실패하면 즉시 절체하지 않는다.

대응 원칙:

1. 실패 항목 기록
2. Firebase 운영은 유지
3. preview/staging에서 수정 후 재리허설

## 6. 리허설 성공 후 다음 단계

아래가 끝나면 절체를 진행한다.

- [ ] 리허설 체크 항목 전체 통과
- [ ] 치명적 에러 로그 없음
- [ ] 롤백 절차 확인

그 다음 순서:

1. 운영 절체 시점 결정
2. Vercel production을 운영 기준선으로 승격
3. Firebase는 즉시 삭제하지 않고 대기
4. 절체 후 1일 모니터링

## 7. 권장 운영 메시지

운영 절체 당일 내부 기준 메시지는 다음 정도면 충분하다.

`Vercel production 리허설이 완료되었고, 현재 운영 기능 기준 회귀는 통과했습니다. 절체 후에도 Firebase 롤백 경로는 유지합니다.`
