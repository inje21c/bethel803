## 1. 인증 스택 개요

- **Supabase Auth** 단일 사용 (자체 토큰/세션 구현 없음)
- 프론트는 `@supabase/supabase-js`의 기본 PKCE 플로우 사용
- 인증 수단 3종 병행 (어느 것도 강제하지 않음):

| 수단 | 용도 | 비고 |
|------|------|------|
| 이메일 + 비밀번호 | 기존 기본 수단 | 유지. 폐지 계획 없음 |
| 구글 OAuth | 비밀번호 분실 문제 해소 | Gmail 가입자는 자동 연결 |
| 카카오 OAuth | 교인 주력 채널(카톡) 대응 | 카톡 인앱 브라우저에서도 동작 |

### 핵심 설계 원칙

1. **login()은 signInWithPassword만 담당.** 프로필 fetch는 `onAuthStateChange(SIGNED_IN)` 단일 경로에 위임 (race condition 방지).
2. OAuth는 `signInWithOAuth`로 리다이렉트만 발생시키고, 복귀 후 처리도 동일하게 `onAuthStateChange`가 담당.
3. public.users 행 생성은 **DB 트리거(handle_new_user)가 전담.** 프론트는 절대 users에 insert하지 않는다.

## 3. 회원가입 플로우

### 3.1 이메일 가입

1. Login.tsx 회원가입 탭: 이름/이메일/비밀번호/구역 입력
2. **구역 미선택 시 가입 차단** (`634c378`). 구역이 1개면 DistrictPicker가 자동 선택
3. `register()`가 `signUp(options.data: { name, district_id })` 호출
4. DB 트리거 `handle_new_user`가 metadata의 district_id로 users 행 생성 (status=pending)
5. 가입 직후 강제 로그아웃 → "구역장 승인 대기" 안내

### 3.2 소셜 가입 (구글/카카오)

OAuth는 리다이렉트 방식이라 가입 시점에 구역을 물을 수 없다.

1. "구글/카카오로 계속하기" → provider 동의 → 복귀
2. `handle_new_user`: metadata에 district_id 없음 → **첫 번째 활성 구역에 임시 배정**, status=pending
3. PendingApproval 화면에서 **구역 확정 강제**:
   - provider가 'email'이 아니고 확정 이력(localStorage `bethel.districtConfirmed.<uid>`)이 없으면 구역 선택 카드가 강조 표시
   - 구역을 능동적으로 선택해야 버튼 활성화 ("이 구역으로 확정")
   - 현 구역과 다른 구역 선택 시 `changeUserDistrict()` (users_update_own RLS로 본인 행 수정)
4. 구역장/마스터 승인 → active

### 3.3 자동 계정 연결 (중요)

- OAuth 로그인 시 **provider가 인증한 이메일이 기존 계정 이메일과 같으면 Supabase가 자동으로 기존 계정에 연결**한다 (새 계정 생성 안 됨)
- 이메일이 다르면 **새 pending 계정이 생긴다** → 기존 사용자는 아래 수동 연결 경로 권장

## 4. 계정 연결 (Profile)

- `supabase.auth.linkIdentity({ provider })` — 로그인된 상태에서 소셜 계정을 현재 계정에 연결
- **Supabase 대시보드 Authentication → Settings → "Allow manual linking" ON 필수.** 꺼져 있으면 "계정 연결 기능이 아직 활성화되지 않았습니다" 토스트
- 연결 후에는 이메일이 달라도 해당 소셜 계정으로 본인 계정 로그인 가능
- 연결 상태는 `supabase.auth.getUserIdentities()`로 조회해 표시
- 연결 해제(unlinkIdentity)는 미구현 (필요 시 추가)

## 5. 비밀번호 복구 경로 3종

| 경로 | 대상 | 구현 |
|------|------|------|
| 이메일 재설정 링크 | 이메일 확인 가능한 사용자 | resetPasswordForEmail → /reset-password (PKCE PASSWORD_RECOVERY 이벤트 + implicit hash 폴백) |
| **관리자 임시 비밀번호** | 이메일 확인 불가/잠긴 사용자 | AdminDashboard → 비밀번호 초기화 → Edge Function이 service role로 변경 → 임시 비밀번호를 카톡 등으로 전달 |
| 소셜 로그인 전환 | 소셜 연결된(또는 같은 이메일) 사용자 | 비밀번호 자체가 불필요해짐 |

### "비밀번호를 잊으셨나요?" 다이얼로그 (752af8f)

로그인 화면의 버튼이 위 3개 경로를 우선순위대로 안내하는 전용 다이얼로그를 연다:

1. **비밀번호 없이 로그인** — 구글/카카오 버튼 (같은 이메일 자동 연결 또는 기연결 계정이면 즉시 해결)
2. **이메일 재설정** — 입력칸(로그인 폼 이메일 프리필) → 발송 후 안내 박스(스팸함 안내)
   + **60초 재발송 쿨다운** (Supabase rate limit 대응, "once every"/"rate limit" 에러를 한국어로 변환)
3. **최후 수단 안내** — 구역장(관리자) 임시 비밀번호 발급 요청

주의: Supabase는 미가입 이메일에도 성공으로 응답한다(계정 존재 노출 방지). 따라서 "메일이
안 와요" 문의는 (a) 스팸함 (b) 가입 이메일 불일치 순으로 확인하고, 안 되면 경로 3으로 안내한다.

### admin-reset-password Edge Function 명세

- 인증: Authorization Bearer(사용자 JWT) → public.users 조회 → **role='master'만 허용**
- 제약: 본인 불가(프로필 사용 안내), master 계정 불가
- 입력: `{ userId, newPassword? }` — newPassword 생략 시 8자 자동 생성 (혼동 문자 0/O/1/l/I 제외)
- 처리: `auth.admin.updateUserById(userId, { password })`
- 출력: `{ ok, userName, tempPassword }`
- 배포: Supabase 대시보드에 코드 직접 붙여넣기 (JWT 검증 ON 유지, 별도 secret 불필요)

## 9. 타교회 확장 시 변경 지점

23_타교회_확장_수익화_계획.md의 Phase 10과 연동. 인증 관점에서 바뀌어야 할 것:

1. **handle_new_user 하드코딩 제거**: cmhyun@gmail.com / bethel803leader@gmail.com 이메일 분기를 없애고 데이터 기반(초대/승격)으로 전환
2. **slug 가입 링크 (`/join/:slug`)**: 가입 시 교회를 URL로 식별 → DistrictPicker는 해당 교회 구역만 노출 (현재 districts anon SELECT는 전 교회 노출이므로 slug 필터 필요)
3. **OAuth 가입의 교회 귀속**: OAuth는 metadata를 못 실으므로, `/join/:slug`에서 slug를 localStorage/쿼리에 보존했다가 PendingApproval 구역 확정 단계에서 해당 교회 구역으로 제한하는 방식 권장
4. **첫 가입자 자동 master(021) 보안**: 빈 교회에 가입하면 master가 되는 로직은 가입 링크 노출 시 교회 탈취 위험 → 초대 토큰 방식으로 교체 필요
5. **admin-reset-password의 교회 경계**: 현재 master는 전체 사용자 대상. 멀티테넌트에서는 `is_church_master(target.church_id)` 검증으로 교체
6. **021 트리거 pending 예외** (8장 참조) — prod 적용 전 필수
7. 카카오/구글 OAuth 앱은 서비스 공통 1개로 충분 (redirect는 Supabase 콜백이므로 교회 수와 무관)
