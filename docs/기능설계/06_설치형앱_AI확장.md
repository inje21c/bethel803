# 설치형 앱 및 AI 확장 전략

## 1. 문서 목적

이 문서는 `bethel803`의 다음 확장 단계인 설치형 웹앱 전환, 구독형 알림, AI 기능 확장 방향을 정리한다.

이 문서는 다음을 목표로 한다.

- 브랜드와 설치형 앱 방향을 먼저 고정한다.
- 웹 설치형 앱과 웹푸시 구현 범위를 단계적으로 정의한다.
- AI 기능을 운영 리스크가 낮은 순서로 도입한다.
- 현재 운영 중인 서비스에 무리 없이 붙일 실행 순서를 정한다.

## 2. 확장 목표

현재 `bethel803`는 핵심 운영 기능이 대부분 구현된 상태다. 다음 단계의 목표는 단순 기능 추가가 아니라, 사용 빈도와 재방문성을 높이는 생활형 공동체 앱으로의 전환이다.

핵심 목표는 다음 3가지다.

1. 설치형 앱으로 전환해 홈 화면 진입성과 재방문성을 높인다.
2. 구독형 알림으로 공동체 운영 이벤트를 더 즉시 연결한다.
3. AI 기능으로 묵상, 설교, 기도, 성경읽기 경험을 더 적극적으로 지원한다.

## 3. 브랜드 결정 사항

### 3.1 앱 이름

- 앱 이름: `벧엘구역`
- 설명 문구: `벧엘교회 구역모임 앱`

판단 근거:

- 실제 사용자 맥락과 정확히 맞는다.
- 설치형 앱 이름으로 짧고 즉시 이해된다.
- 플랫폼형 추상명보다 현재 단계의 채택률에 유리하다.

브랜드 원칙:

- 전면 브랜드는 `벧엘구역`
- 기관 설명은 보조 문구로 둔다

즉, 메인 표기는 `벧엘구역`, 서브 설명은 `벧엘교회 구역모임 앱` 구조로 간다.

### 3.2 앱 아이콘

메인 아이콘 방향은 책 아이콘이 아니라 `벧` 또는 `ㅂ` 기반 모노그램으로 전환한다.

권장 방향:

- 형태: 둥근 사각형
- 배경: 딥 네이비
- 심볼: 흰색 `벧` 또는 단순화한 `ㅂ`
- 스타일: 굵고 단정한 모노그램

1순위 추천:

- `ㅂ`을 열린 책 느낌이 아주 약하게 품은 모노그램

설계 원칙:

- 브랜드 식별성은 모노그램이 담당한다.
- 말씀 중심성은 형태 안에 은근히 포함한다.
- 열린 책 아이콘은 앱 메인 아이콘이 아니라 내부 기능 아이콘으로만 사용한다.

### 3.3 브랜드 성격

한 문장 정의:

`벧엘구역은 구역의 한 주를 연결하고 정리하는 공동체 앱이다.`

브랜드가 가져야 할 인상:

- 신뢰감 있는
- 단정한
- 따뜻하지만 과하지 않은
- 말씀 중심이지만 종교 장식에 기대지 않는
- 구역장이 쓰는 관리툴이 아니라 구역원이 함께 쓰는 앱

브랜드가 피해야 할 인상:

- 통제 중심의 관리 시스템
- 올드한 교회 홈페이지 느낌
- 과한 금색, 과한 십자가, 과한 장식
- 차갑고 공격적인 스타트업 SaaS 느낌

### 3.4 시각 방향

기본 시각 방향:

- 중심색: 딥 네이비
- 배경: 웜그레이 또는 아이보리 그레이
- 포인트: 뮤트 골드 또는 샌드 오렌지 1톤만 제한적으로 사용

원칙:

- 포인트 색은 많게 쓰지 않는다.
- "예배당 게시판"보다 정돈된 모바일 앱 느낌으로 간다.
- "행정 시스템"보다 공동체 도구 느낌을 우선한다.

### 3.5 언어 방향

메인 설명 문구 후보:

- `벧엘교회 구역모임 앱`
- `말씀과 기도, 일정을 한 곳에`
- `구역의 한 주를 더 단정하게`

톤 원칙:

- 과한 은혜 표현보다 정돈된 안내
- 관리 명령문보다 공동체 중심 문장
- 기능 나열보다 한 주의 흐름을 말하는 방식

## 4. 설치형 웹앱 방향

## 4.1 목표

웹 브라우저 접근형 서비스에서 멈추지 않고, 홈 화면에 설치 가능한 생활형 앱으로 전환한다.

핵심 목표:

- 홈 화면 진입
- 앱처럼 실행
- 알림 권한 유도
- 이후 웹푸시 구독 기반 확보

## 4.2 범위

1차 설치형 범위:

- PWA manifest
- 앱 아이콘 세트
- 서비스워커 등록
- 설치 유도 UI
- 기본 오프라인 fallback

2차 확장 범위:

- 웹푸시 구독/해지
- 설치 후 알림 권한 요청 흐름
- 구독 상태 관리 UI

## 4.3 사용자 흐름

권장 흐름:

1. 사용자가 웹앱 진입
2. 설치 가능 조건 충족 시 "설치하기" 노출
3. 설치 후 "구독하기"를 별도 단계로 제안
4. 구독 동의 후 알림 종류 선택

원칙:

- 설치와 구독을 한 번에 강요하지 않는다.
- 설치 직후 곧바로 과도한 권한 요청을 하지 않는다.
- 알림 구독은 가치가 분명한 화면에서 안내한다.

## 5. 구독형 알림 전략

## 5.1 기본 원칙

- 기존 앱 내 알림을 버리지 않는다.
- `notifications`를 단일 진실 공급원으로 유지한다.
- 웹푸시는 앱 내 알림의 외부 전달 채널로 추가한다.
- 알림은 구역 단위 분리를 유지한다.
- 구역원은 본인 소속 구역 알림만 받는다.
- 앱 서비스 운영 관련 공지는 별도 관리자 권한에서 발송할 수 있게 한다.

즉, 구조는 아래처럼 간다.

```text
이벤트 발생
 -> notifications 기록
 -> 구독자 조회
 -> 웹푸시 발송
 -> 발송 결과 로그 저장
```

## 5.2 수신 대상 정책

기본 정책은 다음과 같다.

- 구역원 대상 알림은 각 구역별 회원에게만 발송한다.
- 다른 구역 알림은 수신하지 않는다.
- 앱 서비스 운영 공지는 앱 서비스 관리자만 발송 가능하게 한다.
- 운영 공지는 구역 단위 공지와 별도 성격으로 관리한다.

권한 원칙:

- 구역 운영 알림: 구역 단위
- 앱 서비스 운영 알림: 서비스 전체 단위 또는 관리자 지정 대상

## 5.3 1차 알림 대상

1차 MVP에 포함할 알림:

- 일정 등록 알림
- 참여 여부를 묻는 일정 등록 알림
- 성경공부 발행 알림
- 오늘의 묵상 게시 알림
- 신규 기도제목 알림
- 주간 성경읽기 리마인더 알림

권장 운영 방식:

- 성경읽기는 개별 등록 즉시 알림이 아니라 정기 스케줄 알림으로 운영한다.
- 과도한 빈도의 알림은 구독 해지율을 높일 수 있으므로 사용자의 반복 입력 행동을 전부 즉시 발송으로 연결하지 않는다.

## 5.4 권장 알림 정책

| 알림 유형 | 초기 권장 정책 |
|------|------|
| 일정 등록 | 구역 전체 즉시 발송 |
| 일정 등록 + 참여 확인 필요 | `새 일정이 등록되었습니다. 참여 여부를 알려주세요.` 형태로 구역 전체 발송 |
| 성경공부 발행 | 구역 전체 즉시 발송 |
| 오늘의 묵상 게시 | 구역 전체 또는 사용자 구독 기준 즉시 발송 |
| 신규 중보기도제목 | 구역 전체 즉시 발송 |
| 성경읽기 | 매주 일요일 아침 정기 발송 |
| 앱 서비스 운영 공지 | 앱 서비스 관리자만 발송 가능 |

### 성경읽기 정기 알림 정책

성경읽기 알림은 즉시형이 아니라 주간 스케줄형으로 운영한다.

권장 메시지 형태:

- `지난주(일요일~토요일) 읽은 장수는 00장입니다. 추가할 장수가 있으면 알려주세요.`

권장 발송 시점:

- 매주 일요일 아침

향후 확장:

- 일일 읽기 챌린지
- 목표 장수 설정
- 목표 달성 독려 알림

## 6. AI 기능 확장 전략

## 6.1 AI 확장 원칙

- 운영 기반보다 AI를 먼저 붙이지 않는다.
- 입력 품질이 높은 기능부터 시작한다.
- 요약과 정리처럼 가치가 즉시 보이는 기능부터 도입한다.
- 추천 기능은 실제 운영 데이터와 충돌하지 않게 보조 기능으로 시작한다.
- 1차 AI는 `추천 찬양`부터 시작한다.

## 6.2 1차 AI MVP

1차 MVP의 최우선 기능은 아래와 같다.

### 6.2.1 상황별 찬양 추천

대상 상황:

- 아침 QT
- 기도 시간
- 성경읽기

출력 예시:

- 추천 이유
- 추천 찬양 3곡
- 상황별 재생 분위기 가이드

주의:

- 실제 존재하는 곡과 연결되도록 해야 한다.
- 초기에는 "추천 후보" 개념으로 제공하고, 곡 출처 연동은 단계적으로 붙인다.

판단:

- AI가 사용자 체감 가치를 가장 빠르게 줄 수 있다.
- 운영 서비스에 직접적인 데이터 리스크가 적다.

### 6.2.2 수동 업로드 기반 설교/예배 요약 정리

이 기능은 1차 우선순위 다음 단계로 두되, 초기 구현 범위는 명확히 제한한다.

입력 방식:

- 구역장 또는 운영자가 주일설교, 수요예배, 비정기 설교 음성 파일을 외부 STT 도구에 넣는다.
- 예: 클로바노트에서 텍스트를 추출한다.
- 추출된 텍스트를 이 서비스에 수동 입력 또는 붙여넣기 한다.

출력 목표:

- 잘 정돈된 설교 요약
- 핵심 포인트
- 적용점
- 게시 가능한 초안 형태

발행 흐름:

1. 운영자 또는 구역장이 텍스트 입력
2. AI가 요약 초안 생성
3. 전 구역 게시 가능한 ready 상태로 생성
4. 각 구역장이 수정 후 발행하거나 그대로 발행

원칙:

- 처음부터 음성 업로드 -> 자동 STT -> 자동 발행까지 한 번에 가지 않는다.
- 초기에는 수동 STT 입력 기반으로 품질과 운영 흐름을 먼저 검증한다.

## 6.3 2차 AI 확장

### 6.3.1 성경 암송 기능

가능 기능:

- 오늘의 암송 구절 제안
- 반복 학습 카드
- 빈칸 채우기
- 음성 암송 확인
- 개인 암송 기록

### 6.3.2 기도 도우미

가능 기능:

- 기도제목 분류
- 주간 기도제목 요약
- 기도문 초안 보조

### 6.3.3 성경읽기 보조

가능 기능:

- 본문 요약
- 오늘 본문 핵심 문장
- 적용 질문
- 본문 분위기에 맞는 찬양 추천

## 6.4 AI 출력 원칙

- AI가 최종 진실처럼 보이지 않게 한다.
- 추천/요약/초안임을 분명히 표시한다.
- 실제 신학적 판단이나 공동체 공식 결정은 사용자가 최종 확인한다.

## 7. 구현 우선순위

권장 구현 순서:

1. 설치형 앱 기본 구조
2. 웹푸시 구독 인프라
3. 일정/성경공부/오늘의 묵상/중보기도 푸시 알림
4. 성경읽기 주간 스케줄 알림
5. 상황별 찬양 추천
6. 수동 입력 기반 설교/예배 요약 AI
7. 성경 암송 기능

## 8. 현재 서비스 운영을 고려한 개발 원칙

- 운영 중인 서비스이므로 큰 구조 변경을 한 번에 배포하지 않는다.
- 문서 변경과 설계 반영을 먼저 끝낸다.
- 기능은 작은 묶음으로 나눠서 배포한다.
- 알림 기능은 내부 알림과 웹푸시를 분리해서 단계적으로 확장한다.
- AI 기능은 기존 핵심 흐름을 방해하지 않는 보조 기능으로 시작한다.

## 9. 1차 실행 묶음 제안

### 묶음 A: 설치형 앱 기반

- 앱 이름/설명 반영
- 아이콘 시스템 추가
- manifest 추가
- 설치 버튼 추가
- 기본 서비스워커 도입

### 묶음 B: 구독 알림

- 구독 정보 저장
- 구독/해지 UI
- 일정/성경공부/오늘의 묵상/중보기도 푸시
- 참여 확인 일정 알림 문구 분기
- 성경읽기 주간 스케줄 알림
- 앱 서비스 운영 공지 발송 구조
- 발송 로그 저장

### 묶음 C: AI MVP

- QT/기도/성경읽기 찬양 추천
- 텍스트 스크립트 입력
- 설교/예배 요약 결과 생성
