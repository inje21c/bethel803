### 1.3 SaaS(타교회) 작업 현황 — 이미 착수됨

- `churches` 테이블 스키마 존재: `slug`, `status`, `plan`, `billing_status` 컬럼 포함
  → 과금/수동청구(billing_status='manual')까지 고려된 설계
- RLS 기반 테넌트 격리: `get_my_church_id()`, `is_church_master()`, `is_church_leader()` 등
  헬퍼 함수로 DB 레벨 격리 → **프론트 코드는 church를 모름** (현재 코드에 church 참조 0건)
- `docs/staging_second_church_test.sql`: staging에 제2 교회 생성 스모크 테스트 스크립트 (미커밋)
- `docs/prod_supabase_parity_check.sql`: prod 절체 전 staging/prod 스키마 비교 스크립트 (미커밋)

**즉, 멀티테넌트는 "staging 검증 단계"이며 prod 적용 전이다.**

### 1.4 리스크 / 기술 부채

1. **021 마이그레이션 파일이 repo에 없음.** SaaS 스키마(churches)가 대시보드 SQL로만
   적용된 것으로 보임 → 형상관리 이탈. `supabase/migrations/021_saas_churches.sql`로
   반드시 repo에 회수해야 함.
2. **Edge Functions가 church-aware인지 미확인.** fetch-devotional, parse-bulletin,
   push-dispatch, scheduled-notifications가 단일 교회 전제로 작성됨
   (주보 URL이 bethel.or.kr 하드코딩, 묵상 소스가 sum.su.or.kr 고정).
3. **회원가입 플로우가 단일 교회 전제.** 타교회 사용자가 어느 교회/구역으로
   가입하는지 선택 경로 없음 (slug 기반 가입 링크 필요).
4. **테스트 커버리지 얇음.** api.ts 2,776줄, AdminDashboard 1,755줄 대비 테스트 18개.
   테넌트 격리는 보안 사안이므로 RLS 격리 테스트가 필수.
5. **Edge Function 배포가 대시보드 수동 붙여넣기.** 교회 수가 늘면 운영 불가
   → Supabase CLI access token 기반 GitHub Actions 배포로 전환 필요.
6. **문서 stale.** 05_개발로드맵.md가 푸시/PWA를 "미구현"으로 표기 (실제 완료).
7. **개인정보 이슈 (법적).** 교인 데이터는 개인정보보호법상 **민감정보(종교적 신념)**.
   타교회 서비스 제공 시 개인정보 처리방침, 수탁자 지위 정리, 동의 절차가 선행 조건.

## 3. 수익화 / 후원 모델

### 3.1 비용 구조 (변동비 파악)

| 항목 | 성격 | 비고 |
|------|------|------|
| Vercel | 고정 (Hobby 무료 → Pro $20/월) | 교회 수와 거의 무관 |
| Supabase | 준고정 (Free → Pro $25/월) | DB 용량/MAU 따라 증가 |
| OpenAI API | **변동** | 묵상 요약·주보 파싱이 교회 수에 비례 |
| 웹푸시 | 무료 (자체 VAPID) | |

핵심: **AI 기능만이 교회 수에 비례하는 원가**. 따라서 AI 기능이 유료화의 자연스러운 경계.

### 3.2 권장 모델: 하이브리드 (무료 기본 + 후원 + AI 유료)

종교 서비스 특성상 공격적 과금은 채택 장벽이자 정서적 거부감. 단계적으로:

**1단계 (출시~초기 5교회): 전액 무료 + 후원 안내**
- 모든 기능 무료, 앱 내 "후원하기" 링크 (토스아이디/카카오페이 송금 링크)
- `churches.plan='free'`, `billing_status='manual'` 그대로 사용
- 목적: 채택 사례 확보, 운영 부담 검증

**2단계 (제품 안정 후): Free / Standard 2단 플랜**

| | Free | Standard (월 1~2만원 또는 연 10~20만원) |
|---|---|---|
| 구역 수 | 2개 | 무제한 |
| 기본 기능 (성경공부/기도/읽기/QT/일정) | 전부 | 전부 |
| AI 기능 (주보 파싱, 묵상 요약) | 월 횟수 제한 | 무제한 |
| 웹푸시 | O | O |
| 지원 | 커뮤니티 | 우선 지원 |

- 결제: 초기엔 **계좌이체 수동 청구** (`billing_status='manual'` 설계 그대로),
  교회 10곳 이상부터 토스페이먼츠 정기결제 검토
- 교회는 세금계산서/기부금영수증 관행이 있으므로 수동 청구가 오히려 현실적

**3단계 (선택): 후원 상시화**
- 무료 사용 교회에도 자발적 후원 경로 유지
- 개인 후원(교인)과 교회 후원(기관) 분리 안내

### 3.3 비영리/기부 모델로 갈 경우

- 전 기능 무료 + 운영비 모금 (오픈컬렉티브 또는 국내 후원 플랫폼)
- 장점: 채택 최대화, 교회 간 추천 용이. 단점: AI 원가가 사용량에 비례해 적자 위험
- 절충: **비영리 선언 + AI 기능만 원가 기반 실비 청구** (예: 주보 파싱 월 N회 초과분)

### 3.4 의사결정 기준

- 5교회 이전: 무료 + 후원 (1단계) — 지금 결정할 것은 이것뿐
- 플랜 분기(2단계)는 실제 AI 원가 데이터(월 OpenAI 청구액 / 교회 수)를 본 뒤 결정
- 가격은 "교회 주보 인쇄비보다 싸게"가 심리적 기준선
