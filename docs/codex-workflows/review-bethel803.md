# review-bethel803

## 1. 목적

이 문서는 `bethel803`의 코드 변경을 Codex가 일관된 기준으로 검토하기 위한 실행 문서다.

이 workflow의 목적은 다음과 같다.

- 기능 구현 이후 회귀 위험을 찾는다.
- 운영 환경에서 실제로 문제가 될 수 있는 버그를 우선적으로 식별한다.
- 테스트 누락, 문서 누락, 설정 누락을 함께 검토한다.
- "코드는 돌아가지만 운영에서 깨지는 변경"을 조기에 차단한다.

## 2. 언제 사용하는가

다음 상황에서 사용한다.

- 기능 구현 직후
- 버그 수정 직후
- Auth, 권한, 멀티구역, Supabase 관련 변경 후
- 배포 전 최종 코드 점검
- 대규모 리팩터링 후

## 5. 선행 조건

- 리뷰 대상 변경 범위가 정리되어 있어야 한다.
- 워크트리에 사용자 미관련 변경이 섞여 있으면 구분해야 한다.
- 테스트를 실행했다면 결과를 함께 확인한다.
- 테스트를 실행하지 못했다면 그 사실을 기록한다.

## 6. 리뷰 원칙

- 코드 스타일보다 버그, 리스크, 회귀 가능성을 우선한다.
- "문제가 없을 수도 있음"보다 "깨질 가능성이 있으면 지적"을 우선한다.
- 운영 영향이 큰 변경일수록 더 보수적으로 본다.
- 추정과 확인을 구분한다.
- 발견사항은 재현 가능한 형태로 정리한다.

## 7. 중점 리뷰 대상

`bethel803`에서는 아래 영역을 우선 점검한다.

### 7.1 권한/역할

- `master`, `leader`, `member`, `pending` 흐름이 깨지지 않는가
- `/admin`, `/districts` 접근 제어가 유지되는가
- 클라이언트 UI 차단과 서버/DB 권한이 어긋나지 않는가

### 7.2 멀티구역 스코프

- 쿼리에 `district_id` 누락이 없는가
- 구역 전환 시 이전 구역 데이터가 섞이지 않는가
- 관리자 기능이 잘못 전체 구역에 적용되지 않는가

### 7.3 Supabase 연동

- 테이블/컬럼명과 타입이 코드와 맞는가
- 마이그레이션 없이 코드만 바뀐 부분은 없는가
- Edge Function 호출과 클라이언트 기대 응답이 일치하는가

### 7.4 자동화/운영

- cron 전제 조건이 바뀌었는데 문서가 그대로인 부분은 없는가
- `fetch-devotional`, `parse-bulletin`, `compute_weekly_report()` 영향 범위가 명확한가
- 운영 실패 시 fallback 또는 에러 노출이 충분한가

### 7.5 사용자 경험 회귀

- 로딩, 빈 상태, 에러 상태가 악화되지 않았는가
- 대시보드와 관리자 화면에서 체감 회귀가 없는가
- 기존 경고/안내 문구가 사라지지 않았는가

### 7.6 테스트와 문서

- 변경에 맞는 테스트가 있는가
- 테스트가 너무 좁아 실제 회귀를 놓치지 않는가
- 운영 문서나 체크리스트 갱신이 필요한데 빠지지 않았는가

## 8. 실행 순서

### 8.1 변경 범위 읽기

먼저 아래를 본다.

- 변경된 파일
- 핵심 diff
- 관련 테스트 변경
- 관련 문서 변경

리뷰 시작 시 한 줄로 범위를 요약한다.

예시:

- 대시보드 묵상 fallback 표시 추가
- 멀티구역 주간 마감 파라미터 반영
- 관리자 성경읽기 기간 조회 기능 추가

### 8.2 고위험 변경 여부 판단

아래 중 하나라도 해당하면 고위험 변경으로 본다.

- Auth
- role
- district scope
- Supabase schema
- Edge Function
- cron
- Storage 공개 정책

고위험이면 이 항목들을 우선 본다.

- 회귀 가능성
- 운영 설정 누락
- 문서 미반영

### 8.3 기능적 정확성 검토

변경이 의도한 동작을 실제로 만족하는지 본다.

확인 질문:

- 입력값 경계가 처리되는가
- null/empty/fallback 케이스가 안전한가
- 비동기 실패 시 UI가 깨지지 않는가
- 기존 흐름을 덮어쓰지 않는가

### 8.4 데이터/스코프 검토

특히 Supabase 쿼리와 관리자 기능은 아래를 본다.

- 필요한 필터가 빠지지 않았는가
- district scope가 유지되는가
- 권한이 약한 사용자도 우회 호출 가능한 구조가 아닌가
- optimistic update 또는 캐시 무효화가 맞는가

### 8.5 테스트 검토

아래를 점검한다.

- 새 동작에 대한 테스트가 있는가
- 실패 케이스 테스트가 있는가
- 기존 테스트가 실제 변경을 검증하는가
- 테스트가 없어도 되는 변경인지 설명 가능한가

### 8.6 문서/운영 영향 검토

아래 중 하나라도 해당하면 문서 반영 필요로 본다.

- 운영 절차가 바뀜
- 점검 체크리스트가 바뀜
- 환경 변수 또는 배포 순서가 바뀜
- 사용자 역할 기대 동작이 바뀜

### 8.7 최종 리뷰 판정

최종 판정은 아래 중 하나로 한다.

- `문제 없음`
- `수정 권장`
- `수정 필요`

기준:

- 운영 회귀 위험이 없고 테스트/문서도 맞으면 `문제 없음`
- 당장 치명적이지 않지만 품질 저하 가능성이 있으면 `수정 권장`
- 배포 전 해결해야 할 버그/리스크가 있으면 `수정 필요`

## 9. 결과 작성 형식

리뷰 결과는 아래 순서로 작성한다.

### 9.1 Findings

발견사항이 있으면 심각도 높은 순서로 쓴다.

형식:

- 심각도
- 파일/라인
- 문제 설명
- 왜 위험한지
- 권장 수정 방향

### 9.2 Open Questions

확신할 수 없는 부분이 있으면 질문이나 가정으로 분리한다.

### 9.3 Summary

마지막에 짧게 요약한다.

- 전체 판단
- 남은 리스크
- 테스트/문서 상태

## 10. 발견사항 분류 기준

### 10.1 Critical

- 권한 우회
- 다른 구역 데이터 노출
- 데이터 손실 가능성
- 운영 자동화 오작동 가능성

### 10.2 High

- 핵심 기능 실패 가능성
- 관리자 기능 회귀
- Edge Function 호출 불일치
- 배포 전 반드시 수정해야 하는 설정/문서 누락

### 10.3 Medium

- fallback/에러 처리 부족
- 일부 화면 회귀
- 테스트 누락
- 문서 불일치

### 10.4 Low

- 유지보수성 저하
- 불필요한 중복
- 읽기 어려운 구조

## 11. 완료 조건

이 workflow는 아래 조건을 만족하면 완료다.

- 변경 범위를 읽고 핵심 리스크를 검토함
- 발견사항을 심각도 순으로 정리함
- 테스트/문서 관점까지 검토함
- 최종 판정을 남김

## 12. 차단 시 처리

다음 경우 완전 리뷰 대신 제한 리뷰로 마무리한다.

- diff가 너무 커서 전체 문맥이 부족함
- 원격 환경 의존으로 실제 동작을 확인할 수 없음
- 테스트 계정이나 데이터가 없음

이 경우 반드시 아래를 남긴다.

- 확인한 범위
- 확인 못한 범위
- 리뷰 신뢰도를 낮추는 조건
- 추가 확인이 필요한 지점
