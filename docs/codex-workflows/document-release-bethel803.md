# document-release-bethel803

## 1. 목적

이 문서는 `bethel803`에서 코드 변경 이후 문서와 릴리스 상태를 정리하기 위한 실행 문서다.

이 workflow의 목적은 다음과 같다.

- 코드 변경과 문서 상태를 동기화한다.
- 운영 절차 변경이 문서에 반영되지 않는 문제를 줄인다.
- 배포 또는 운영 인수인계 시 필요한 정보를 빠뜨리지 않는다.
- "코드는 바뀌었지만 문서는 과거 상태"인 상황을 방지한다.

## 2. 언제 사용하는가

다음 상황에서 사용한다.

- 기능 개발 완료 후
- 운영 절차 변경 후
- 배포 전후
- 자동화 경로 변경 후
- 권한/역할/환경 변수/체크리스트가 바뀐 후

## 4. 입력 정보

실행 전 아래 정보를 정리한다.

- 실제 변경 파일 목록
- 실행한 테스트/검증 결과

- 배포 필요 여부

- 구현 또는 수정 작업이 끝나 있어야 한다.
- 최소한의 검증 결과가 있어야 한다.
- 변경 범위가 문서 관점에서 어느 영역에 해당하는지 판단되어야 한다.

## 6. 문서화 원칙

- 문서는 변경 이후의 사실만 적는다.
- 아직 확인되지 않은 운영 사실은 단정하지 않는다.
- 로컬 확인과 원격 확인을 구분한다.
- 사용자가 다음 작업을 이어갈 수 있게 쓴다.
- 사소한 코드 세부사항보다 운영 영향과 사용 방법을 우선한다.

## 8. 실행 순서

### 8.1 변경 요약 작성

먼저 이번 변경을 아래 네 줄로 정리한다.

- 무엇이 바뀌었는가
- 왜 바뀌었는가
- 사용자 영향이 있는가
- 운영 영향이 있는가

### 8.2 문서 영향 범위 판정

아래 질문으로 문서 범위를 결정한다.

- 사용 방법이 바뀌었는가
- 운영 순서가 바뀌었는가
- 환경 변수/배포 전제가 바뀌었는가
- 체크리스트 항목이 바뀌었는가
- 역할 기대 동작이 바뀌었는가

### 8.3 대상 문서 선정

최소 1개 이상의 문서를 갱신한다.

권장 규칙:

- 사용자 기능 변경이면 사용자-facing 문서 1개
- 운영 변경이면 운영 문서 1개 이상
- 고위험 변경이면 체크리스트 문서까지 함께 갱신

### 8.4 문서 수정

수정 시 아래 원칙을 따른다.

- 장황한 설정보다 실제 필요한 항목만 적는다.
- 오래된 설명은 남기지 않는다.
- 추후 확인이 필요한 내용은 체크 항목이나 메모로 남긴다.
- 날짜와 현재 상태를 명확히 적는다.

### 8.5 릴리스 노트 수준 요약 작성

배포 여부와 상관없이 아래를 짧게 정리한다.

- 변경 영역
- 핵심 사용자/운영 영향
- 검증 결과
- 남은 리스크

### 8.6 후속 작업 연결

아래 중 하나라도 남아 있으면 후속 항목으로 연결한다.

- 원격 미확인 항목
- 운영자가 수동으로 해야 하는 작업
- 다음 배포 전에 해결해야 할 항목
- 문서상 판단만 있고 실제 실행이 남은 항목

## 11. 차단 시 처리

다음 경우 문서화를 보류하지 말고 제한 상태로라도 남긴다.

- 원격 환경을 확인할 수 없음
- 배포 여부가 아직 미정
- 운영 담당자 확인이 필요함

이 경우 반드시 아래를 남긴다.

- 현재까지 확정된 사실
- 아직 미확정인 사실
- 누구 확인이 필요한가
- 문서에 임시로 남겨야 할 경고/메모

## 14. 메모

- `bethel803`는 운영 문서 비중이 높아서 문서 반영이 릴리스의 일부다.
- 문서화는 "좋으면 하는 것"이 아니라 "현재 상태를 다음 사람에게 넘기는 일"이다.
- 특히 cron, Edge Function, Storage, 역할별 권한은 반드시 문서와 같이 관리한다.
