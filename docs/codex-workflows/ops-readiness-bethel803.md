# ops-readiness-bethel803

## 1. 목적

이 문서는 `bethel803`의 운영 준비 상태를 Codex가 일관된 순서로 점검하기 위한 실행 문서다.

이 workflow의 목적은 다음과 같다.

- 코드 구현 완료와 운영 활성화 완료를 구분한다.
- 로컬에서 확인 가능한 항목과 원격 환경에서 별도 확인이 필요한 항목을 분리한다.
- 점검 결과를 운영 준비 보고서 형식으로 남긴다.

## 2. 언제 사용하는가

다음 상황에서 사용한다.

- 운영 시작 전 최종 점검
- 배포 직전 상태 확인
- Supabase 설정 변경 후 재점검
- Edge Function, cron, Storage 관련 변경 직후
- 운영 이슈가 보고되어 현재 준비 상태를 재평가할 때

## 4. 입력 정보

실행 전 아래 정보를 확보한다.

- 현재 작업 범위
- 최근 변경 파일 목록
- 배포 대상 환경 구분
  - 로컬
  - staging
  - production
- 사용할 Supabase 프로젝트 식별 정보
- 검증 가능한 URL 또는 운영 접근 경로

원격 접근이 필요한 경우 추가로 필요하다.

- Supabase 대시보드 접근 권한
- 배포 URL
- 테스트 계정 또는 운영 점검 계정

## 5. 선행 조건

- 저장소 최신 상태를 확인한다.
- 미커밋 변경이 있으면 이번 점검 대상인지 구분한다.
- 로컬 검증 명령이 실행 가능한지 확인한다.
- 원격 검증 권한이 없으면 그 사실을 먼저 보고서에 명시한다.

## 6. 점검 원칙

- 확인한 것과 추정한 것을 구분한다.
- 로컬에서 확인 가능한 항목은 실제로 확인한다.
- 원격 환경이 필요한 항목은 "미확인"으로 넘기지 말고, 확인에 필요한 조건과 명령 또는 절차를 남긴다.
- 운영 관련 변경이 있었다면 문서 동기화를 완료 조건에 포함한다.
- 위험 항목은 "문제 없음"보다 "추가 확인 필요"로 보수적으로 분류한다.

## 7. 실행 순서

### 7.1 범위 정리

먼저 이번 점검이 무엇을 위한 것인지 한 줄로 정리한다.

예시:

- 운영 시작 전 전체 준비 점검
- Edge Function 배포 후 운영 재점검
- cron 설정 변경 후 자동화 경로 재검증

이후 점검 범위를 아래 5개 묶음으로 나눈다.

1. 프론트 환경 변수
2. Supabase 기본 구성
3. Storage / Edge Functions / cron
4. 역할별 운영 기능
5. 배포 및 문서 상태

### 7.2 로컬 코드 기준 점검

아래 항목은 저장소 기준으로 먼저 확인한다.

- 환경 변수 키가 코드와 문서에서 일치하는가
- 마이그레이션 파일이 존재하는가
- Edge Function 코드가 존재하는가
- 운영 문서가 현재 구조를 반영하는가
- 체크리스트 문서가 최신 상태인가

권장 확인 파일:

- `src/lib/supabase.ts`
- `src/lib/authContext.tsx`
- `supabase/migrations/*`
- `supabase/functions/*`
- `docs/OPERATIONS.md`
- `docs/운영준비체크리스트.md`
- `docs/운영점검TODO.md`

### 7.3 로컬 검증 명령 실행

최소 아래 명령을 실행한다.

```bash
npm test
npm run build
```

필요 시 추가한다.

- 번들 크기 확인
- 특정 테스트 파일 재실행
- 타입/린트 점검

실행 결과는 아래처럼 기록한다.

- 성공
- 실패
- 미실행

실패 시:

- 실패한 명령
- 핵심 에러 원인
- 운영 영향도

를 같이 남긴다.

### 7.4 운영 문서 기준 점검

문서에서 요구하는 운영 전제 조건을 표로 정리한다.

필수 점검 항목:

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`
- `VITE_APP_URL`
- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `OPENAI_API_KEY`
- `attachments` 버킷
- `fetch-devotional` 배포
- `parse-bulletin` 배포
- `weekly-close-all-districts` 등록
- 주보 PDF 파싱 스케줄 등록

### 7.5 원격 확인 항목 분리

아래 항목은 로컬 코드만으로 확정할 수 없다.

- 실제 Supabase 환경 변수 설정 여부
- Storage 버킷 실제 존재 여부
- Edge Function 실제 배포 여부
- cron 실제 등록 여부
- 운영 URL 실제 동작 여부

따라서 각 항목을 아래 셋 중 하나로 분류한다.

- 확인 완료
- 확인 필요
- 접근 권한 부족

그리고 `확인 필요` 또는 `접근 권한 부족` 항목에는 반드시 다음을 남긴다.

- 누가 확인해야 하는가
- 어떤 화면 또는 명령으로 확인하는가
- 확인되면 어떤 상태가 정상인가

### 7.6 기능 운영 준비 점검

아래 기능이 운영 기준으로 준비되었는지 확인한다.

- 인증
- 성경공부
- 기도제목
- 성경읽기
- 일정 및 출석
- 관리자 기능
- 알림 및 검색
- 자동화 기능

이 단계에서는 기능 자체 구현 여부보다 운영 준비 관점으로 본다.

예시:

- 첨부 업로드 기능이 코드상 존재하는가
- 업로드 버킷이 실제 있어야만 동작하는가
- 자동화 함수는 있어도 스케줄이 없으면 운영 준비 완료로 볼 수 없는가

### 7.7 문서 동기화 판단

점검 중 아래 중 하나라도 발견되면 문서 갱신 대상으로 잡는다.

- 실제 운영 경로와 문서가 다름
- 실패 cron 또는 제거 대상이 문서에 반영되지 않음
- 체크리스트와 현재 코드 구조가 다름
- 신규 운영 리스크가 발견되었는데 문서에 없음

### 7.8 최종 분류

최종 상태는 아래 셋 중 하나로 판정한다.

- `운영 가능`
- `조건부 운영 가능`
- `운영 준비 미완료`

판정 기준:

- 핵심 운영 전제 조건이 모두 확인되면 `운영 가능`
- 원격 확인만 일부 남고 코드/문서/검증이 정렬되면 `조건부 운영 가능`
- 배포, 권한, 자동화, Storage 중 핵심 전제가 비어 있으면 `운영 준비 미완료`

## 8. 산출물 형식

결과는 아래 형식으로 정리한다.

### 8.1 요약

- 점검 목적
- 점검 일시
- 점검 환경
- 최종 판정

### 8.2 상태 표

| 구분 | 항목 | 상태 | 근거 | 후속 조치 |
|------|------|------|------|----------|

상태는 아래 값만 사용한다.

- 완료
- 확인 필요
- 권한 부족
- 실패

### 8.3 핵심 리스크

- 운영 전에 해결해야 하는 것
- 조건부로 넘어갈 수 있는 것
- 문서만 보강하면 되는 것

### 8.4 권장 다음 단계

1. 즉시 필요한 조치
2. 운영 직전 필요한 조치
3. 운영 후 안정화 조치

## 9. 완료 조건

이 workflow는 아래 조건을 만족하면 완료다.

- 로컬 코드/문서 기준 점검이 끝남
- 가능한 검증 명령이 실행됨
- 원격 필요 항목이 명시적으로 분리됨
- 최종 판정이 내려짐
- 후속 조치가 우선순위와 함께 정리됨

## 10. 실패 또는 차단 시 처리

다음 경우 workflow를 중단하지 말고 상태 보고로 마무리한다.

- 원격 권한이 없음
- 운영 URL 접근이 안 됨
- Supabase 대시보드 확인 불가
- 테스트 계정이 없음

이 경우 반드시 아래를 남긴다.

- 현재까지 확인한 사실
- 아직 확인하지 못한 항목
- 추가로 필요한 권한 또는 정보
- 다음 담당자가 이어서 확인할 순서

## 11. bethel803 기본 점검 순서

실무에서는 아래 순서를 기본으로 사용한다.

1. `docs/운영점검TODO.md` 읽기
2. `docs/운영준비체크리스트.md` 읽기

4. `npm test`

7. 원격 확인 필요 항목 분리
8. 최종 운영 준비 보고서 작성

## 12. 메모

- `bethel803`는 코드 구현과 운영 활성화를 별도로 봐야 한다.
- 자동화 기능은 "코드 있음"과 "실제 스케줄 동작"을 구분해야 한다.
- 운영 준비 점검은 "문서가 맞는가"까지 포함해야 한다.
