# bethel803 아키텍처 설계서

**프로젝트명:** 벧엘교회 킨텍스장성남 구역 관리 시스템  
**작성일:** 2026-03-12  
**대상 버전:** React SPA + Supabase 기반 운영 구조

## 1. 문서 목적

이 문서는 bethel803의 시스템 아키텍처를 정의한다. 기능 목록이 아니라, 다음 항목을 명확히 하는 것이 목적이다.

- 어떤 구성요소가 어떤 책임을 가지는지
- 프론트엔드, 데이터베이스, 인증, 자동화가 어떻게 연결되는지
- 현재 구현 상태와 목표 구조가 어디서 만나는지
- 향후 기능 추가 시 어떤 규칙을 따라야 하는지

## 2. 아키텍처 목표

### 2.1 비즈니스 목표

- 구역원과 구역장이 같은 데이터를 여러 기기에서 일관되게 사용한다.
- 주간 운영 업무를 수동 입력 중심에서 자동 집계 중심으로 전환한다.
- 개인 신앙 데이터는 본인 중심으로 보호하고, 운영 데이터는 구역장이 관리한다.
- 주간 마감, 묵상 수집, 주보 PDF 처리 같은 반복 업무를 자동화한다.

### 2.2 기술 목표

- 프론트엔드는 단순 UI 렌더링이 아니라 인증 상태와 데이터 권한을 일관되게 반영한다.
- 데이터 저장은 `localStorage`가 아니라 Supabase DB를 단일 진실 공급원으로 사용한다.
- 권한 제어는 화면 분기만이 아니라 DB 레벨 정책까지 포함한다.
- 자동화 기능은 Edge Functions와 스케줄 작업으로 분리한다.
- 기능별 확장은 가능하되, 공통 인프라 계층은 재사용 가능해야 한다.

## 3. 전체 시스템 구조

```text
[Browser / React SPA]
  - React Router
  - AuthContext
  - TanStack Query
  - UI Components
  - Feature Pages
          |
          v
[Application Access Layer]
  - src/lib/api.ts
  - src/lib/authContext.tsx
  - src/lib/supabase.ts
          |
          v
[Supabase Platform]
  - Auth
  - PostgreSQL
  - Row Level Security
  - Storage
  - Edge Functions
          |
          +--> [Scheduled Jobs / CRON]
          |      - daily devotional fetch
          |      - weekly close
          |      - bulletin parse
          |
          +--> [External Services]
                 - OpenAI API
                 - 두란노/매일성경 묵상 페이지
                 - 벧엘교회 주보 PDF
```

## 4. 레이어 설계

## 4.1 프레젠테이션 레이어

구성 요소:

- `src/pages/*`: 화면 단위 페이지
- `src/components/*`: 공통 UI 및 운영 컴포넌트
- `src/components/AppLayout.tsx`: 공통 레이아웃
- `src/App.tsx`: 라우팅, 전역 Provider 조립

책임:

- 사용자 입력 수집
- 인증 상태에 따른 라우트 제어
- 서버 데이터 표시
- 낙관적 상호작용이 아닌, 현재는 조회/저장 결과 기반 UI 갱신

비책임:

- DB 스키마 판단
- 권한의 최종 보장
- 배치 작업 실행

## 4.2 애플리케이션 접근 레이어

구성 요소:

- [src/lib/authContext.tsx](/home/ubuntu/bethel803/src/lib/authContext.tsx)
- [src/lib/api.ts](/home/ubuntu/bethel803/src/lib/api.ts)
- [src/lib/supabase.ts](/home/ubuntu/bethel803/src/lib/supabase.ts)

책임:

- 인증 세션 감시
- Supabase SDK 호출 래핑
- 페이지에서 사용할 View Model 수준의 타입 변환
- 공통 예외 전달

설계 원칙:

- 페이지는 가능하면 Supabase를 직접 호출하지 않고 `api.ts` 또는 인증 컨텍스트를 통해 접근한다.
- DB 컬럼명과 UI 속성명이 다르면 접근 레이어에서 변환한다.
- 권한 분기는 UI 편의용으로만 두고, 실제 보호는 RLS에 의존한다.

## 4.3 데이터 레이어

구성 요소:

- Supabase PostgreSQL
- SQL migration (`supabase/migrations/*.sql`)
- RLS 정책

주요 역할:

- 사용자 프로필
- 성경공부 및 답안
- 기도 제목
- 성경 읽기 로그
- 일정 및 출석
- 묵상 콘텐츠
- 주간 보고서
- 알림

데이터 레이어 원칙:

- 개인 데이터는 `user_id`를 기준으로 소유권을 명확히 가진다.
- 집계성 데이터는 별도 스냅샷 테이블에 저장한다.
- 자동화 산출물은 원본 데이터와 분리된 테이블로 관리한다.

## 4.4 자동화 및 통합 레이어

구성 요소:

- `supabase/functions/fetch-devotional`
- `supabase/functions/parse-bulletin`
- 향후 추가될 weekly close function

책임:

- 외부 사이트 스크래핑
- OpenAI 호출
- 스케줄 실행
- 주간 마감 처리와 집계 생성

분리 이유:

- 브라우저에서 외부 스크래핑이나 비밀키 처리 불가
- 장시간/정기 실행 작업은 사용자 요청 흐름과 분리 필요
- 운영 실패 시 재시도와 로그 추적이 가능해야 함

## 5. 현재 구현 아키텍처

현재 코드 기준 핵심 구조는 다음과 같다.

### 5.1 프론트엔드 진입

- [src/App.tsx](/home/ubuntu/bethel803/src/App.tsx): `QueryClientProvider`, `TooltipProvider`, `BrowserRouter`, `AuthProvider`, `ErrorBoundary`를 조립한다.
- React Query는 5분 `staleTime`, 포커스 복귀 시 자동 재조회 비활성화, 재시도 1회로 설정되어 있다.

### 5.2 인증 흐름

- [src/lib/authContext.tsx](/home/ubuntu/bethel803/src/lib/authContext.tsx)에서 `supabase.auth.onAuthStateChange`로 세션 상태를 구독한다.
- 로그인 후 `users` 테이블에서 역할과 상태를 읽는다.
- `pending` 사용자는 `/pending`으로 이동하고, `leader`만 관리자 라우트 접근이 가능하다.

### 5.3 데이터 접근

- [src/lib/api.ts](/home/ubuntu/bethel803/src/lib/api.ts)가 `bible_studies`, `study_answers`, `prayer_requests`, `bible_reading_logs`, `schedules`, `attendances` 등 주요 테이블 접근을 담당한다.
- 이 레이어에서 snake_case DB 컬럼을 camelCase UI 모델로 변환한다.

### 5.4 인프라 연결

- [src/lib/supabase.ts](/home/ubuntu/bethel803/src/lib/supabase.ts)는 브라우저용 Supabase 클라이언트와 DB 타입 정의를 제공한다.
- 운영 구조는 이미 localStorage-only 단계를 넘어 Supabase 전환 중인 상태다.

## 6. 목표 아키텍처

## 6.1 컨텍스트 다이어그램

```text
사용자(구역원/구역장)
   |
   v
React SPA
   |
   +--> Supabase Auth
   +--> Supabase Postgres
   +--> Supabase Storage
   +--> Supabase Edge Functions
                 |
                 +--> OpenAI API
                 +--> 외부 묵상 사이트
                 +--> 교회 주보 PDF
```

## 6.2 핵심 컴포넌트

| 컴포넌트 | 역할 |
|---------|------|
| React SPA | 사용자 UI, 상태 표시, 입력 처리 |
| AuthContext | 세션 관리, 사용자 프로필 로딩, 역할 판별 |
| API Layer | 기능별 데이터 조회/저장, 타입 매핑 |
| Supabase Auth | 로그인, 회원가입, 비밀번호 재설정 |
| PostgreSQL | 핵심 업무 데이터 영속 저장 |
| RLS | 사용자/구역장 데이터 권한 제어 |
| Edge Functions | 스크래핑, AI 호출, 배치 처리 |
| Storage | 향후 첨부파일 및 PDF 저장 |
| CRON | 정기 작업 트리거 |

## 7. 기능별 데이터 흐름

## 7.1 로그인

```text
사용자 입력
 -> AuthContext.login()
 -> Supabase Auth signInWithPassword
 -> onAuthStateChange
 -> users 프로필 조회
 -> role/status 반영
 -> 화면 라우팅
```

## 7.2 성경공부 답안 저장

```text
BibleStudyDetail
 -> api.saveStudyAnswer()
 -> study_answers upsert
 -> 사용자별 답안 저장
 -> 완료 상태 반영
```

보안 원칙:

- 본인 답안만 조회 가능
- 구역장은 완료 여부 중심 집계만 조회
- 답안 내용 자체는 타인에게 노출하지 않음

## 7.3 기도 제목 응답

```text
PrayerRequests
 -> api.getPrayerRequests()
 -> prayer_requests + users 조인 조회
 -> 구역장 응답 저장
 -> answered 상태 갱신
```

## 7.4 일정 및 출석

```text
ScheduleManagement
 -> schedules CRUD
 -> attendances upsert
 -> 일정별 출석 현황 조회
 -> 주간 보고서 집계 대상 데이터 축적
```

## 7.5 오늘의 묵상 자동화

```text
CRON
 -> fetch-devotional function
 -> 외부 묵상 페이지 수집
 -> OpenAI 요약 생성
 -> daily_devotionals 저장

```

## 7.6 주보 PDF 자동화

```text
CRON

```

## 9. 권한 및 보안 설계

## 9.1 권한 모델

| 역할 | 범위 |
|------|------|
| member | 본인 데이터 입력/조회, 공용 발행 데이터 조회 |
| leader | member 권한 + 운영 데이터 조회/관리 + 승인/발행 |

추가 상태:

| 상태 | 의미 |
|------|------|
| pending | 가입 대기, 주요 기능 접근 불가 |
| active | 정상 사용 가능 |

## 9.2 보안 원칙

- 화면 숨김만으로 권한을 보장하지 않는다.
- 모든 개인 데이터는 RLS로 최종 제한한다.
- 비밀키는 브라우저에 두지 않고 Edge Function 환경변수에 둔다.
- OpenAI 호출은 서버 측 Function에서만 수행한다.
- 자동화 입력값은 원본 URL, 생성 시각, 실행 로그를 남길 수 있어야 한다.

## 10. 장애 및 운영 설계

## 10.1 장애 대응 원칙

- 프론트는 API 실패 시 토스트 또는 오류 화면으로 명확히 알려야 한다.
- 인증 상태 로딩은 무한 대기하지 않도록 타임아웃 방어가 필요하다.
- 자동화 작업은 재시도 정책과 수동 재실행 경로를 가져야 한다.

## 10.2 관측 포인트

- 인증 실패 건수
- Edge Function 실행 성공/실패
- 주간 마감 작업 성공 여부
- 외부 스크래핑 실패율
- OpenAI 호출 실패율/응답 시간

## 10.3 백업 및 복구

- DB는 Supabase 백업 정책을 따른다.
- 주간 보고서는 계산 결과를 스냅샷으로 저장해 원본 변경과 분리한다.
- 자동 생성된 성경공부는 발행 전 검토 단계를 둔다.

## 11. 배포 아키텍처

## 11.1 프론트엔드

- Vite 빌드 산출물을 정적 호스팅한다.
- 환경변수:
  - `VITE_SUPABASE_URL`
  - `VITE_SUPABASE_ANON_KEY`
  - `VITE_APP_URL`

## 11.2 백엔드/플랫폼

- Supabase 프로젝트가 Auth, DB, Storage, Function 실행 환경을 제공한다.
- 마이그레이션은 `supabase/migrations` 기준으로 관리한다.
- Edge Function 배포는 기능 단위로 분리한다.

## 12. 개발 원칙

### 12.1 코드 구조 원칙

- 페이지는 UI와 사용자 상호작용에 집중한다.
- 데이터 접근 로직은 `src/lib/api.ts` 또는 기능별 API 모듈로 분리한다.
- 공통 권한 로직은 `AuthContext`와 RLS에 집중시킨다.
- 테이블 수가 증가하면 `src/lib/api.ts`를 도메인별 파일로 분리한다.

### 12.2 확장 원칙

- 새 기능은 가능한 기존 도메인에 귀속시킨다.
- 자동화가 필요한 기능은 브라우저 코드에 억지로 넣지 않는다.
- 통계/보고는 원본 트랜잭션 테이블과 스냅샷 테이블을 분리한다.
