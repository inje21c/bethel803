## 한 줄 요약
베타 오픈 첫날 발생한 오유입 사고를 계기로, 3단계(교회→구역→구역원) 구조를 **모임우선**으로 재설계. 일반 사용자는 커뮤니티 컨테이너 밑에 "모임"을 만들고, 교회는 슈퍼어드민 B2B 전용으로 격하. 설계부터 prod 배포·검증·운영 도구까지 완료. 마이그레이션 **052~059**, 커밋 **d80486e ~ 321b1b2**.

## 배경 (사고)
- 2026-06-20 베타 오픈. 사용자가 교회 등록 대신 구역원으로 가입 → 전역 "첫 활성 구역"(테스트구역)에 자동 배정되는 테넌트 경계 침범.
- 원인 2곳: (a) `get_active_districts_with_church` RPC + `DistrictPicker`가 anon에 전 테넌트 구역 열거, (b) `handle_new_user`의 `ORDER BY created_at LIMIT 1` 전역 fallback.

## 채택 아키텍처
- **커뮤니티 컨테이너 교회**(고정 UUID `…0002`, plan=community, master 없음, trial 없음) 밑에 일반 사용자 모임(district)을 두고 개설자=leader.
- 리더는 기존 RLS(leader-구역스코프)로 자기 모임만 봄 → 격리 신규 코드 0.
- 교회=슈퍼어드민이 B2B로 생성하는 희소 단위. 모임→교회 승격 경로 제공.
- 컨테이너에 master를 두지 않음 → 콘텐츠 프라이버시 + 통삭제 방지.

## 발견·교훈
- **공유 컨테이너 + church 단위 unique 제약**: `districts_church_name_unique` 때문에 동명 모임 가입이 깨짐 → 056 de-dup. 다른 per-church 제약도 공유 컨테이너에서 재검토 필요.
- **컨테이너 master 부재의 양면**: 프라이버시·삭제안전 보장하지만, 역할 변경 등 운영이 막힘 → 슈퍼어드민 전용 RPC(트리거 우회)로 해소(059).
- **트리거 우회 패턴**: SECURITY DEFINER + `ALTER TABLE … DISABLE/ENABLE TRIGGER USER`로 prevent_district_church_change/enforce_role_change 등 가드 우회(052/058/059). service-role REST로는 불가 → 마이그레이션(DDL)으로만.
- **배포 순서**: 프론트가 052 컨테이너에 의존 → prod DB 먼저, 그다음 main 머지. 짧은 degraded 창 주의.
