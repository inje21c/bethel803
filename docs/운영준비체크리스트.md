# bethel803 운영 준비 체크리스트

**프로젝트명:** 벧엘교회 킨텍스장성남 구역 관리 시스템  
**작성일:** 2026-03-25  
**기준:** 현재 저장소 코드 상태 기준 운영 전 점검 항목

## 1. 목적

이 문서는 `bethel803`를 실제 운영 환경에 올리기 전에 확인해야 할 항목을 정리한다.

- 코드 구현 완료와 운영 활성화 완료를 구분한다.
- 필수 인프라 설정 누락을 방지한다.
- 운영 시작 전 우선순위를 빠르게 판단할 수 있게 한다.

## 2. 현재 상태 요약

코드 기준으로 다음 항목은 이미 구현되어 있다.

- Supabase Auth 기반 로그인, 회원가입, 승인 대기, 비밀번호 재설정
- Supabase DB 기반 성경공부, 기도제목, 성경읽기, 일정, 출석
- 관리자 기능: 구역원 승인/권한 변경, 성경공부 CRUD, 주간 보고, CSV 내보내기
- Edge Function 기반 묵상 수집, 주보 PDF 파싱
- 앱 내 알림, 전역 검색, 일정 첨부 업로드

다만 다음 항목은 코드 존재와 실제 운영 활성화 여부를 별도로 확인해야 한다.

- Supabase 프로젝트 환경 변수 연결
- Edge Function 배포
- `pg_cron` 스케줄 등록
- Storage 버킷 생성
- PWA 실제 활성화 여부

현재 운영 메모:

- `fetch-devotional`은 외부 실행 경로에서 정상 동작한 기록이 확인되었다.
- Supabase DB cron `fetch-devotional-daily`는 실패 상태가 확인되어 제거 대상으로 판단했다.
- 주간 마감은 단일 구역 cron `weekly-close`를 제거하고 멀티 구역 cron `weekly-close-all-districts`로 전환했다.

## 3. 최우선 점검 항목

### 3.1 프론트엔드 환경 변수

운영 배포 환경에 아래 값이 설정되어 있어야 한다.

| 키 | 용도 |
|----|------|
| `VITE_SUPABASE_URL` | 브라우저용 Supabase URL |
| `VITE_SUPABASE_ANON_KEY` | 브라우저용 Supabase anon key |
| `VITE_APP_URL` | 비밀번호 재설정 리디렉션 URL |

확인 근거:

- `src/lib/supabase.ts`
- `src/lib/authContext.tsx`

### 3.2 Supabase Edge Function 환경 변수

아래 값이 Supabase Functions 환경에 설정되어 있어야 한다.

| 키 | 용도 |
|----|------|
| `SUPABASE_URL` | 서비스 클라이언트 연결 |
| `SUPABASE_SERVICE_ROLE_KEY` | service role 권한 호출 |
| `OPENAI_API_KEY` | OpenAI API 호출 |

대상 함수:

- `supabase/functions/fetch-devotional`
- `supabase/functions/parse-bulletin`

### 3.3 Edge Function 배포 확인

실제 배포 여부를 확인해야 한다.

- `fetch-devotional`
- `parse-bulletin`

확인 항목:

- 함수가 Supabase 프로젝트에 배포되어 있는가
- 최신 코드 버전이 반영되었는가
- 함수 로그에서 최근 에러가 없는가

### 3.4 Cron 등록 확인

코드와 마이그레이션에는 예시 SQL이 있으나, 실제 등록 여부는 별도 확인이 필요하다.

확인 대상:

- 매주 일요일 주간 마감 집계
- 매주 일요일 20:00 KST 주보 PDF 파싱

운영 방침:

- 묵상 수집은 현재 외부 실행 경로를 유지한다.
- 실패 중인 Supabase DB cron `fetch-devotional-daily`는 제거한다.
- 주간 마감은 `weekly-close-all-districts` 기준으로 운영한다.

현재 확인된 스케줄:

- `weekly-close-all-districts`
- `0 6 * * 0` (매주 일요일 15:00 KST)

주의:

- 현재 저장소 기준 `cron.schedule(...)` 예시는 주석 상태다.
- 실제 운영에서는 Supabase 대시보드 또는 SQL Editor에서 등록해야 한다.

### 3.5 Storage 버킷 생성 확인

일정 첨부 업로드 기능은 `attachments` 버킷을 전제로 한다.

확인 항목:

- `attachments` 버킷이 생성되어 있는가
- Public 접근 정책이 운영 정책과 일치하는가
- 업로드 후 공개 URL이 정상 동작하는가

## 4. 기능별 운영 확인

### 4.1 인증

확인 항목:

- 회원가입 후 `pending` 상태로 생성되는가
- 구역장 승인 후 정상 로그인되는가
- 비밀번호 재설정 메일 링크가 운영 URL로 돌아오는가
- `master`, `leader`, `member` 권한이 화면과 DB 정책에서 일치하는가

### 4.2 구역 관리

확인 항목:

- 활성 구역 목록이 정상 조회되는가
- 마스터가 구역 생성/수정/비활성화 가능한가
- 신규 가입 시 `district_id`가 올바르게 저장되는가

### 4.3 성경공부

확인 항목:

- 발행된 자료만 구역원에게 노출되는가
- 답변 저장/수정/완료가 정상 동작하는가
- 관리자에서 자료 수동 등록/수정/삭제가 가능한가
- 주보 PDF 파싱 결과가 `published=false`로 들어오는가

### 4.4 기도제목

확인 항목:

- 일반 구역원은 본인 및 공유 허용 범위에 맞는 항목만 조회되는가
- 구역장은 구역 단위 운영 화면을 정상 조회하는가
- 중보기도 수 집계가 정상 반영되는가

### 4.5 성경읽기

확인 항목:

- 일별 기록 저장이 정상 동작하는가
- 누적 합계가 사용자별/구역별로 맞는가
- 관리자 통계 조회가 기간 필터와 함께 정확한가

### 4.6 일정 및 출석

확인 항목:

- 일정 CRUD가 구역 단위로 분리되는가
- 출석 응답이 사용자별로 1건씩 upsert 되는가
- 일정 첨부 업로드와 URL 노출이 정상 동작하는가

### 4.7 관리자 기능

확인 항목:

- 구역원 승인/거절/권한 변경 가능 여부
- 주간 보고서 조회 및 수동 마감 실행 가능 여부
- CSV 다운로드 동작 여부
- 접속 정보(`last_login_at`)가 누적되는가

### 4.8 알림

확인 항목:

- 구역장/마스터만 알림 발행 가능한가
- 구역 단위로 알림이 분리되는가
- 읽음 처리 후 뱃지 수가 감소하는가

## 5. 데이터 및 권한 점검

### 5.1 마이그레이션 적용 확인

운영 DB에 아래 핵심 변경이 모두 반영되어 있어야 한다.

- `districts` 테이블
- `district_id` 컬럼 및 관련 FK
- `notifications`, `notification_reads`
- `prayer_intercessions`
- `source_pdf_url`
- `last_login_at`
- `compute_weekly_report()` 함수

### 5.2 RLS 점검

확인 항목:

- 구역원은 타 구역 데이터를 조회할 수 없는가
- 구역장은 자기 구역 범위만 관리 가능한가
- 마스터는 전체 구역 운영이 가능한가
- 익명 접근이 남아 있지 않은가

### 5.3 샘플 계정 점검

최소 아래 테스트 계정으로 검증한다.

- `master` 1명
- `leader` 1명
- `member` 2명 이상
- 서로 다른 구역 소속 사용자

## 5.4 현재 즉시 수행할 정리

- 실패 중인 `fetch-devotional-daily` cron 제거
- 외부 실행 경로가 실제 운영 주체임을 문서에 반영
- `weekly-close-all-districts` 재등록 및 수동 검증 반영

## 6. 배포 점검

### 6.1 프론트 배포

현재 저장소에는 Firebase Hosting 설정이 포함돼 있다.

확인 항목:

- `npm run build` 성공
- SPA rewrite 동작
- `dist` 기준 정적 배포 정상
- 캐시 헤더 정책 적절성

### 6.2 빌드 품질

현재 빌드는 통과하지만 관리자 화면 청크가 크다.

권장 확인:

- 초기 진입 속도
- 관리자 화면 로딩 체감
- 모바일 네트워크에서의 성능

필요 시 다음 작업을 별도 수행한다.

- `AdminDashboard` 탭 단위 분할
- 차트 및 무거운 관리자 의존성 lazy loading
- production minify 정책 재검토

## 7. 미완료 또는 재판단 필요 항목

### 7.1 PWA

패키지는 포함되어 있으나 현재 설정상 PWA가 실제 활성화된 것으로 보기 어렵다.

현 상태:

- `vite-plugin-pwa` 의존성 존재
- `vite.config.ts`에 PWA 플러그인 설정 없음
- SW 등록 코드 없음

판단:

- 운영 범위에 포함할지 결정 필요
- 포함 시 manifest, SW, install flow, offline cache 정책을 별도 구현해야 함

### 7.2 문서 불일치

현재 일부 문서는 여전히 `localStorage` 중심 시범판 상태로 남아 있다.

우선 수정 대상:

- `docs/기능설계/01_서비스개요_현재구현.md`
- `docs/기능설계/04_데이터_아키텍처.md`

## 9. 운영 준비 완료 기준

아래 조건을 모두 만족하면 운영 시작 가능 상태로 본다.

- 프론트 배포 URL에서 로그인/회원가입/재설정 흐름이 정상 동작한다.
- Supabase DB와 RLS가 구역 단위 권한을 올바르게 강제한다.
- `fetch-devotional`, `parse-bulletin`, 주간 마감이 실제 스케줄로 동작한다.
- `attachments` 업로드가 정상 동작한다.
- 관리자 계정으로 승인, 보고서, 알림, 성경공부 관리가 가능하다.
- 문서가 현재 구현 상태와 크게 어긋나지 않는다.
