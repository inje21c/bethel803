# bethel803 개발 진행 순서

**작성일:** 2026-03-09
**기준 문서:** 기능설계서.md (v1.0) + 기능설계서_추가요건.md (v1.1)

## 원칙

- 백엔드 없이 가능한 작업을 먼저 처리
- 다음 단계의 전제 조건이 되는 작업을 선행
- 구역원이 체감하는 핵심 기능 우선
- 자동화(CRON)는 수동 기능이 완성된 후 적용

> 현재 코드베이스에서 바로 적용 가능. 배포 즉시 효과.

| 순서 | 작업 | 대상 파일 | 근거 |
|------|------|----------|------|
| 1-1 | CSS `@import` 순서 오류 수정 | `index.css` | 빌드 경고 제거 |
| 1-2 | Vite 번들 코드 스플리팅 설정 | `vite.config.ts` | 565KB → 분산, 로딩 개선 |
| 1-3 | 성경공부 답안 접근 권한 분리 (A2) | `store.ts`, `BibleStudyDetail.tsx`, `AdminDashboard.tsx` | 프라이버시 즉시 보장 |

| 1-5 | 카카오톡 공지 메시지 생성 UI (A1) | `AdminDashboard.tsx` + `KakaoNoticeGenerator.tsx` 신규 | 실사용 즉시 가능 |
| 1-6 | 다크 모드 토글 버튼 | `AppLayout.tsx` | next-themes 이미 세팅됨 |

## Phase 2 — 백엔드 기반 구축

> 이후 모든 단계의 전제 조건. 가장 중요한 단계.

| 순서 | 작업 | 상세 |
|------|------|------|
| 2-1 | Supabase 프로젝트 생성 | PostgreSQL DB + Auth + Storage + Edge Functions |
| 2-2 | DB 스키마 설계 및 마이그레이션 | `users`, `study_answers`, `prayer_requests`, `bible_reading_logs`, `schedules`, `attendances`, `bible_studies`, `daily_devotionals`, `weekly_reports` |
| 2-3 | Supabase Auth 연동 | 이메일/비밀번호 회원가입·로그인, JWT 토큰, role 필드 |
| 2-4 | 회원가입 요청 → 구역장 승인 흐름 | `pending` 상태 → 승인 시 `active` |
| 2-5 | 비밀번호 재설정 이메일 | Supabase Auth 기본 기능 활용 |
| 2-6 | 프론트엔드 API 클라이언트 교체 | `store.ts` → `supabase-js` 클라이언트로 전환 |
| 2-7 | localStorage 데이터 → DB 마이그레이션 스크립트 | 기존 사용자 데이터 보존 |
| 2-8 | Row Level Security (RLS) 정책 설정 | 본인 데이터만 접근, 구역장 전체 조회 허용 |
