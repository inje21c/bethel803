## 전략 방향 요약

### 핵심 포지셔닝
> "개인 QT 습관 + 소모임 리더 도구"를 하나의 앱으로
> 리더가 쓰면 구성원 전체가 따라온다

### 타겟 우선순위
1. **페르소나 B (소모임 리더)** — 최대 유입 채널. 리더 경험 최우선
2. **페르소나 A (신앙 습관 형성자)** — 리텐션 핵심
3. **페르소나 C (기존 구역원)** — 기존 동선 보존 중요

### 경쟁 차별점
- 바이블애플: 성경읽기 특화, 그룹통독 → 우리는 QT+소모임 통합
- 큐티한스푼: QT 플로우 명확, 소모임 기능 없음 → 우리의 기회
- **우리만의 것**: 구역장이 카톡 없이 구성원을 관리하고 독려할 수 있는 유일한 앱

## 2026-06-19 UX/비즈니스 전문가 분석 결론

### UX 관점 핵심 판단
- Sprint 2 홈 화면 재구성 방향 **유효** — QT 히어로 카드 집중, 하단 탭 4개 구조
- **성경읽기 위치 확정**: 구역(커뮤니티) 서브탭 유지. "개인 독서 도구"가 아니라 "구역 함께 읽기" 맥락으로 포지셔닝
  - Sprint 5 온보딩에서 "우리 구역과 함께 성경 읽기" 의미 부여
  - 기능 수 줄이지 않고 온보딩으로 컨텍스트 제공
- 구역장 대시보드 인지 부담: 섹션 레이블("구역 운영" / "나의 활동")로 충분히 해소 가능

### 비즈니스 관점 핵심 판단
- **후원하기 버튼**: Toss Payments 정식 결제 연동보다 먼저 구현 가능
  - 개인 계좌번호 or 카카오페이/토스 개인송금 링크 — 사업자 등록 불필요
  - Landing 페이지 + Onboarding 페이지에 배치
  - 이용자가 자발적으로 후원하는 모델 (선택적)
- **Toss Payments 정식 연동**: 사업자 등록 완료 후로 연기
- **주보 파싱**: 단일 교회(벧엘) URL 하드코딩 제거 필요 — 다교회 환경 부적합. hasModule 설계 재검토

### Sprint 3 — 계획 (다음)
**리더 허브 + 깨우기 알림**

#### 목표
구역장이 카톡 없이 구성원을 관리·독려할 수 있는 완성형 리더 허브

| 항목 | 우선순위 | 비고 |
|------|---------|------|
| 후원하기 버튼 | P0 | Landing + Onboarding. 카카오페이/토스 링크 or 계좌번호 |
| 구역 탭 리더 전용 섹션: 구성원 QT 현황 | P0 | 오늘/이번 주 완료 여부 한눈에 |
| 깨우기(Nudge) 버튼 | P1 | push-dispatch 연동, 하루 1회 제한 |
| 관리자 대시보드 출석 편집 상세 | P1 | 일정별 출석 현황 + 편집 |
| 주보 파싱 다교회 대응 | P2 | URL 하드코딩 제거, hasModule 재설계 |

#### 후원하기 설계
```
[Landing 페이지]  [Onboarding 페이지]
  ↓ "후원하기" 버튼
  → 카카오페이 개인송금 QR or 링크
     or 토스 계좌 링크
     or 계좌번호 표시 바텀시트
```
- 사업자 불필요 — 개인 계좌/개인 간 이체 형식
- 금액 지정 없는 자발적 후원

#### 깨우기 알림 설계
```
구역장이 "깨우기" 버튼 클릭
  → 오늘 QT 미완료 구역원 리스트 확인
  → 한 번에 푸시 발송
  → 메시지: "오늘 QT 함께해요 :) — {구역장이름}이 기다리고 있어요"
  → 하루 1회 제한 (push_deliveries 체크)
```
- 기존 push-dispatch Edge Function 활용
- 신규 RPC: `get_qt_incomplete_members(district_id, date)` 필요
- 발송 기록: push_deliveries 테이블 (nudge_type 컬럼 추가 or memo 활용)

## 기술 결정 사항 (오늘)

### DistrictPicker 교회명 표시 패턴
- `anon`은 `churches` 테이블 직접 읽기 불가 (034 RLS)
- 해결: `SECURITY DEFINER` RPC `get_active_districts_with_church()` → anon도 JOIN 조회 가능
- **동일 패턴이 필요한 상황**: anon이 교회 정보가 필요한 공개 화면 전반에 적용

### 출석 수동 입력 RLS
- `saveAttendance(scheduleId, userId, status)`: `userId`가 타인이어도 구역장 권한으로 저장 가능
- RLS 정책이 구역장(leader/master)의 구역원 출석 UPSERT를 허용하고 있음 (검증 완료)

### "우리 구역은?" 데이터 신뢰도
- **조건**: 주간보고 마감(`compute_weekly_report`)이 실행되어야 집계 생성
- **전제**: 구역모임 일정 등록 + 출석 체크 프로세스 정착
- 마감이 없는 주는 strip 미표시 (의도된 동작)
